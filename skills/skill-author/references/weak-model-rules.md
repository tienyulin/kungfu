# 弱模型相容規則 — 寫 skill 時把判斷轉成機制

團隊的執行模型能力參差,skill 要寫到**最弱的模型也走得完**。原理:強模型靠判斷,
弱模型靠機制 —— 把每個「要想」的地方換成「照抄 / 二選一 / 跑腳本」。

## 五條硬規則(新 skill 必須全過;改舊 skill 逐步補)

### 1. 自由度階梯 —— 每步選最低可行的自由度

| 自由度 | 形式 | 何時用 |
|---|---|---|
| 低(首選) | 逐字可抄物:完整指令、prompt 區塊、檔案模板 | 順序固定、錯了會壞的步驟 |
| 中 | 骨架 + 佔位符(`<服務名>`) | 有固定模式、參數會變 |
| 高(避免) | 開放式指示(「寫出好摘要」) | 只准用在真的多解皆可處,且**必附 fallback 公式** |

高自由度步驟的 fallback 公式 = 拿不準就照填的樣板(例:摘要寫不好 →
「<這東西做什麼>+<何時會用到>」兩句式)。弱模型抄公式的結果 ≥ 它自由發揮。

### 2. 判斷點一律封閉分流

決策寫成「條件 → 走哪」的二/三選一,條件要**可觀察**(檔案存在?指令輸出含 X?),
不要「視情況」「適當地」。分流蓋不到的情況 → 明寫「都不符 → 停下問使用者」。

### 3. 機器 gate 取代自我判斷

「檢查你的產出是否…」弱模型會自評全過。改成**可跑的驗證**:lint/validator 腳本、
`pytest`、`grep` 斷言 —— 每個關鍵產出至少一道腳本 gate,紅了照錯誤訊息修,
綠了才准進下一步。沒有現成腳本的產出,至少給「逐字對照清單」(產出必含哪些字面值)。

### 4. 進度 checklist 照抄打勾

多步驟 skill 開頭放一個 checklist 區塊,指示「**先把這個清單照抄進你的回覆,
每完成一步打勾再做下一步**」。弱模型中途漂移(跳步、忘記後半)是最常見死法,
checklist 是最便宜的錨。

### 5. 「未定義 = 停」護欄

skill 沒講的情況,弱模型會自信亂編。每個 skill 明寫一句:
「**遇到本 skill 與 references 都沒定義的情況:停下來問使用者,不要自行發明。**」
禁止清單(不得做什麼)比正面指示更有效:弱模型對「不要 X」的遵循度高於「要優雅地 Y」。

### 6. 反捏造:內容溯源 + 驗證附輸出

弱模型撞擊測試實測到的兩種捏造,skill 要各設一道閘:
- **內容捏造**:產出的每個事實(env 名、數字、URL、行為描述)必須有出處
  (repo 可觀察 / 使用者提供);查無 → 寫「待補:<要問什麼>」並問使用者,
  **禁止用「常見做法」補**。產文件類 skill 加自查步驟:「逐項核對每個事實在
  repo/輸入找得到出處」。
- **驗證捏造**:回報只准寫**實際執行過**的驗證並**附指令輸出**;沒跑的寫「未跑」。
  宣稱「全綠」但貼不出輸出 = 沒跑。

## 寫作細節(同樣為弱模型)

- 一句一指令:不要一句塞三個動作;順序敏感就用編號清單。
- 代名詞少用:「它」「該檔」換成明確名稱 —— 弱模型指代消解差。
- 範例給**完整的**輸入→輸出對,不給片段(弱模型不會腦補省略號)。
- 同一概念全篇同一個詞(官方 consistent terminology;弱模型把同義詞當不同東西)。
- 引用只深一層且明說「現在去讀 <檔>」;不要「參見」(弱模型常不跟連結)。

## 驗收:弱模型撞擊測試

新 skill 或大改後,除了 fresh-context 盲審,**用團隊最弱的模型當 executor 實跑一個
小情境**(各 agent 用各自的 subagent／model 選擇機制,例如 Claude Code 的 Agent 工具可指定
model),產出用機器 gate + 強模型/人工驗收。
弱模型跑不完的步驟 = skill 的自由度還太高,回去降。
