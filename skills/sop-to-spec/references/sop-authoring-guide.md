# 怎麼寫一份好的 SOP（給提需求的人）

這份文件教你：把一個人工操作流程寫成 SOP，好讓 `sop-to-spec` 把它變成一份規格、
再交給 AI 照著做出 API。你不用懂程式，只要把**業務上的事**講清楚。

空白骨架在 [sop-template.md](sop-template.md)，複製它照著填。

## 為什麼要照這個格式填

- AI **只看規格**做事，不會回頭猜你心裡想的。
- 規格裡「怎樣算對、怎樣算錯」這些驗收，**全部來自你在 SOP 寫的內容**。
- 你哪一格空著沒寫，AI 就得自己發明一個行為——通常就跟你要的不一樣，產出就歪掉。

所以：**填得越清楚，AI 做出來越接近你要的**。花十分鐘填滿，省掉來回改十次。

## 誰負責什麼（你只管中間這欄）

| 角色 | 負責 | 例子 |
|------|------|------|
| **你（寫 SOP）** | 這隻 API 的內容 | 做什麼、每個參數（名稱/型態/預設值/要檢查什麼）、詳細步驟、危不危險要誰批、失敗怎麼辦、留什麼紀錄 |
| **sop-to-spec skill** | 技術翻譯 | 把你寫的變成 API 規格、補上登入驗證/重複請求/同時操作這類系統機制 |
| **技術審批者** | 補技術缺口 | 回答 skill 產規格時冒出來的技術問題 |

原則：**SOP 沒寫的，開發就不知道**——能寫多詳細就多詳細。參數的名稱、型態、預設值、
要不要先檢查輸入，這些雖然偏開發，但只有你知道業務上該長怎樣，要寫（型態用白話寫就行，
見下面）。只有**純系統機制**不用寫：登入怎麼驗、同時兩個人操作、同一個請求送兩次——
這些系統會自動處理。

## 檔案怎麼放（先建好資料夾再寫）

在專案資料夾底下建 `docs/`，裡面建 `sops/` 和 `specs/` 兩個資料夾。
`sops/` 裡按**主題分組**建資料夾，組名你自己取（`account`、`inventory`…）：

```
<專案>/docs/
├ sops/
│  ├ account/               ← 一組（都在動「帳號」）
│  │  ├ delete-account.md   ← 一隻 API 一份 SOP 檔
│  │  └ query-account.md
│  └ inventory/             ← 另一組（動「庫存」）另開資料夾
└ specs/                    ← 之後工具產的規格會照同樣的分組放這裡
```

- **一隻 API 一份檔**：檔名用那個動作命名（`delete-account.md`）。
- **同一組**（都在動同一個東西）→ 放同一個資料夾。
- **不同組** → 另開資料夾，不要混。
- 拿不準算不算同一組？問自己：「這些 API 是不是都在動同一個東西？」是 → 同組；不是 → 分開。

## 每份 SOP（＝一隻 API）填什麼

一隻 API 裡面**可能有好幾個步驟**（例：刪帳號＝先停用 → 通知使用者 → 刪資料）。
結構是：開始前的確認 → 依序的步驟（每步可帶自己的前後確認）→ 全部做完的確認。

- **名字**（標題，一句話）：例「刪除使用者帳號」。
- **做什麼**（講清楚，不要只寫一句話）：做什麼、給誰用、什麼時候會用到、做了之後對
  系統/業務的影響。例「客服在使用者申請刪除帳號時使用。刪除後帳號不能再登入、個資會清除，
  但訂單紀錄依法規要保留。」
- **輸入**（每個參數一列的表）：**名稱、型態、必填？、預設值、輸入限制/要先檢查什麼、說明**。
  - 型態用你會的話寫：文字／整數／小數／是-否／日期時間／清單。
  - 「輸入限制」寫要不要先檢查值：不能是空的？要是正數？有固定格式（像 Email）？超出範圍怎麼辦？
  - 參數名稱寫不出英文就寫中文，開發會取名。
- **輸出**（每個欄位一列的表）：**名稱、型態、說明**。查詢類＝會查到的欄位**全部列出來**；
  變更類＝做完會回什麼。例「user_id（文字）＝被刪的帳號、deleted_at（日期時間）＝什麼時候刪的」。
- **開始前先確認**（precheck，選填）：動手前業務上要先確認什麼？沒有可留空，
  **有幾條就依序列幾條**。例「操作者要有管理員權限」「這個帳號沒有未結清的訂單」。
- **步驟**（依序，**寫詳細**）：這是整份 SOP 的核心，越詳細 AI 越不用猜。
  人工操作時你怎麼做，就怎麼寫；一步寫不清楚就拆成兩步。
  每一步可以帶自己的「做之前確認」「做完確認」，沒有就不寫。
  例「1. 把帳號狀態改成停用（做完確認：不能再登入） 2. 寄刪除通知信到帳號的 Email
  3. 清除個資欄位，訂單紀錄不動（做完確認：帳號查不到、訂單還查得到）」。
- **全部做完，怎麼算成功**（postcheck，選填）：
  例「帳號查不到了、狀態變已刪除、使用者有收到通知」。
- **危險嗎／能不能反悔／要誰批准**：做了會不會無法復原、要不要人簽核？
  **不能反悔的操作，一定要寫清楚要誰批准、憑什麼批准**（例：變更單號）——這點 skill
  不會替你發明，你不寫就會卡住。
- **可能出什麼錯、現場怎麼處理**（含哪一步出錯怎麼辦）：用你平常的話寫。
  例「查無此帳號→回報找不到；有未結訂單→擋下不刪；通知寄不出去→照樣刪，記下來補寄」。
- **測試例子**（2–3 個）：舉幾個「**這樣輸入 → 應該得到這樣的結果**」，輸入用「輸入」表的參數寫。
  例「user_id=沒訂單的帳號 → 成功，帳號消失」「user_id=有未結訂單的帳號 → 被擋，帳號還在」。
  這幾個例子會直接變成 AI 自我驗證有沒有做對的依據，**很重要，別跳過**。
- **要留什麼紀錄、給誰看**：要記什麼。例「記操作人、時間、刪了哪個帳號」。

## 你填的每一格，會變成規格的什麼

（你不用懂右邊，只是讓你知道每格都有用、不是白填的。）

| 你填的 | 會變成 | 讓 AI 能 |
|--------|--------|---------|
| 做什麼（詳細說明） | API 的用途與範圍 | 知道在做哪件事、為誰做 |
| 輸入表（名稱/型態/預設值/限制） | 參數定義與格式檢查 | 收對資料、擋錯資料 |
| 輸出表（名稱/型態） | 回傳格式 | 回對資料 |
| 開始前先確認 | 前置條件檢查 | 擋掉不該做的操作 |
| 步驟（含每步前後確認） | 執行流程與中途失敗的處理 | 照對的順序做、卡住知道算什麼 |
| 全部做完怎麼算成功 | 成功情境的驗收 | 知道對的樣子 |
| 危險/誰批准 | 風險等級與防護 | 危險操作要通關密語才做 |
| 會出什麼錯 | 失敗情境的驗收與錯誤表 | 知道錯的樣子、怎麼回應 |
| 測試例子 | 驗收準則與測試計畫 | 自己驗證有沒有做對 |
| 要留什麼紀錄 | 稽核紀錄 | 每個操作留 log |

## 填不出來怎麼辦

- **業務判斷填不出來**（例：這動作到底能不能反悔、失敗該怎麼處理）→ 這正是只有你知道的事，
  問清楚業務再填，別留空。
- **參數細節寫不精確**（例：型態不知道怎麼講、名稱想不出英文）→ 用白話寫、用中文取名，
  開發看得懂就好；真的不知道的格子標「請開發補」，別整格空著。
- **純系統機制**（例：登入怎麼驗、要不要防同時操作）→ **不用管**，那是 skill 跟技術審批者的事。
