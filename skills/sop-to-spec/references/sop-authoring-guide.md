# 怎麼寫一份好的 SOP（給提需求的人）

這份文件教你：把一個人工操作流程寫成 SOP，好讓 `sop-to-spec` 把它變成一份規格、
再交給 AI 照著做出 API。你不用懂程式，只要把**業務上的事**講清楚。

空白骨架在 [sop-template.md](sop-template.md)，複製它照著填。

## 為什麼要照這個格式填

- AI **只看規格**做事，不會回頭猜你心裡想的。
- 規格裡「怎樣算對、怎樣算錯」這些驗收，**全部來自你在 SOP 寫的內容**。
- 你哪一格空著沒寫，AI 就得自己發明一個行為——通常就跟你要的不一樣，產出就歪掉。

所以：**填得越清楚，AI 做出來越接近你要的**。花十分鐘填滿，省掉來回改十次。

## 誰負責什麼（你只管中間這欄）

| 角色 | 負責 | 例子 |
|------|------|------|
| **你（寫 SOP）** | 業務判斷 | 有哪些動作、做之前要確認什麼、哪個危險要誰批、失敗怎麼辦、留什麼紀錄 |
| **sop-to-spec skill** | 技術翻譯 | 把你寫的變成 API、判斷要拆成幾個 API、補上登入驗證/格式檢查/重複請求這類系統機制 |
| **技術審批者** | 補技術缺口 | 回答 skill 產規格時冒出來的技術問題 |

**登入、資料格式對不對、同時兩個人操作、同一個請求送兩次會怎樣——這些你不用寫**，
系統會自動處理。你只要專心把「業務上該怎麼運作」講清楚。

## 每一格在問什麼（照這個順序填）

### 整份 SOP 開頭
- **這份 SOP 做什麼**（一句話）：例「把刪除使用者帳號的流程變成 API」。
- **開工前整體確認**（選填）：整組動作開始前，整體要先成立什麼？沒有可留空。
  例「操作者必須有管理員權限」。

### 每個動作，逐一填（有幾個動作就填幾塊）

動作很多、橫跨好幾個主題（帳號＋訂單＋通知…）也沒關係——**全部列出來就好**。
要不要拆成幾個 API 不是你的事，工具會判斷並拿提案跟你確認。

- **做什麼**：這個動作在做的事。例「刪除一個使用者帳號」。
- **輸入**：要提供什麼資料才能做。例「使用者 ID」。
- **輸出**：做完會回傳/得到什麼。查詢類要寫「會查到哪些欄位」；變更類寫「會產生什麼」。
  例「回傳被刪帳號的 ID 與刪除時間」。
- **做之前先確認**（precheck，選填）：這步動手前，業務上要先確認什麼？沒有可留空，
  **有的話依序寫**。例「先確認這個帳號沒有未結清的訂單」。
- **做完怎麼算成功**（postcheck，選填）：怎麼確認這步真的成功了？
  例「帳號在系統裡查不到了、狀態變成已刪除」。
- **危險嗎／能不能反悔／要誰批准**：這動作做了會不會無法復原、要不要人簽核？
  **不能反悔的操作，一定要寫清楚要誰批准、憑什麼批准**（例：變更單號）——這點 skill
  不會替你發明，你不寫就會卡住。
- **可能出什麼錯、現場怎麼處理**：用你平常的話寫。例「查無此帳號→回報找不到；
  帳號還有未結訂單→擋下來、不刪」。

### 整份 SOP 結尾
- **全部做完整體驗什麼**（選填）：整件事跑完，整體要檢查什麼？沒有可留空。
- **怎樣算整件事成功、怎樣算失敗**：一句話講清楚成功與失敗的界線。
- **測試例子**（2–3 個）：舉幾個「**這樣輸入 → 應該得到這樣的結果**」。
  例「刪一個沒有訂單的帳號 → 成功，帳號消失」「刪一個有未結訂單的帳號 → 被擋，帳號還在」。
  這幾個例子會直接變成 AI 自我驗證有沒有做對的依據，**很重要，別跳過**。
- **要留什麼紀錄、給誰看**：每個動作要記什麼。例「記操作人、時間、刪了哪個帳號」。

## 你填的每一格，會變成規格的什麼

（你不用懂右邊，只是讓你知道每格都有用、不是白填的。）

| 你填的 | 會變成 | 讓 AI 能 |
|--------|--------|---------|
| 開工前整體確認 | 進入條件 | 擋住不該進來的請求 |
| 每步「做之前先確認」 | 前置條件檢查 | 擋掉不該做的操作 |
| 每步「做什麼／輸入／輸出」 | API 端點與收送的資料 | 知道收什麼、回什麼 |
| 每步「做完怎麼算成功」 | 成功情境的驗收 | 知道對的樣子 |
| 每步「危險/誰批准」 | 風險等級與防護 | 危險操作要通關密語才做 |
| 每步「會出什麼錯」 | 失敗情境的驗收與錯誤表 | 知道錯的樣子、怎麼回應 |
| 整體成功/失敗＋測試例子 | 驗收準則與測試計畫 | 自己驗證有沒有做對 |
| 要留什麼紀錄 | 稽核紀錄 | 每個操作留 log |

## 填不出來怎麼辦

- **業務判斷填不出來**（例：這動作到底能不能反悔、失敗該怎麼處理）→ 這正是只有你知道的事，
  問清楚業務再填，別留空。
- **技術細節不確定**（例：資料格式、要不要防同時操作）→ **不用管**，那是 skill 跟技術審批者的事。
