# 怎麼寫一份好的 SOP（給提需求的人）

這份文件教你：把一個人工操作流程寫成 SOP，好讓 `sop-to-spec` 把它變成一份**規格**
（＝給開發和 AI 看的詳細說明書），再交給 AI 照著做出 **API**（＝一個可以被系統執行的
功能，例如「刪除帳號」「查詢申請進度」）。你不用懂程式，只要把**業務上的事**講清楚。

空白骨架在 [sop-template.md](sop-template.md)，複製它照著填。

## 為什麼要照這個格式填

- AI **只看規格**做事，不會回頭猜你心裡想的。
- 規格裡「怎樣算對、怎樣算錯」這些驗收，**全部來自你在 SOP 寫的內容**。
- 你哪一格空著沒寫，AI 就得自己發明一個行為——通常就跟你要的不一樣，產出就歪掉。

所以：**填得越清楚，AI 做出來越接近你要的**。花十分鐘填滿，省掉來回改十次。

## 誰負責什麼（你只管中間這欄）

| 角色 | 負責 | 例子 |
|------|------|------|
| **你（寫 SOP）** | 這個功能的內容 | 做什麼、誰能用、每個參數（名稱/型態/預設值/要檢查什麼）、詳細步驟、能不能復原、失敗怎麼辦、留什麼紀錄 |
| **sop-to-spec skill** | 技術翻譯 | 把你寫的變成規格、補上軟體自己該處理的機制 |
| **技術審批者** | 補技術缺口 | 回答產規格時冒出來的技術問題 |

原則：**SOP 沒寫的，開發就不知道**——能寫多詳細就多詳細。參數的名稱、型態、預設值、
要不要先檢查輸入，這些雖然偏開發，但只有你知道業務上該長怎樣，要寫（型態用白話寫就行，
見下面）。只有**軟體自己會處理的事**不用寫：怎麼確認操作者的身分（登入）、兩個人同時
操作怎麼排隊、同一個請求不小心送兩次怎麼擋——這些系統會自動處理。

## 檔案怎麼放（先建好資料夾再寫）

在專案資料夾底下建 `docs/`，裡面建 `sops/` 和 `specs/` 兩個資料夾。
`sops/` 裡按**主題分組**建資料夾，組名你自己取（`account`、`inventory`…）：

```
<專案>/docs/
├ sops/
│  ├ account/               ← 一組（都在動「帳號」）
│  │  ├ delete-account.md   ← 一個功能一份 SOP 檔
│  │  └ query-account.md
│  └ inventory/             ← 另一組（動「庫存」）另開資料夾
└ specs/                    ← 之後工具產的規格會照同樣的分組放這裡
```

- **一個功能一份檔**：檔名用那個動作命名（`delete-account.md`）。
- **同一組**（都在動同一個東西）→ 放同一個資料夾。
- **不同組** → 另開資料夾，不要混。
- 拿不準算不算同一組？問自己：「這些功能是不是都在動同一個東西？」是 → 同組；不是 → 分開。

## 每份 SOP（＝一個功能）填什麼

一個功能裡面**可能有好幾個步驟**（例：刪帳號＝先停用 → 通知使用者 → 刪資料）。
結構是：開始前的確認 → 依序的步驟（每步可帶自己的前後確認）→ 全部做完的確認。
照範本的節順序填：

- **名字**（標題，一句話）：例「刪除使用者帳號」。
- **做什麼**（講清楚，不要只寫一句話）：做什麼、給誰用、什麼時候會用到、做了之後對
  系統/業務的影響。
- **誰可以用**：哪些角色能用，清單列**完整**（不要寫「等」）；系統要不要強制擋沒權限的人。
- **特別名詞或分類**（選填）：SOP 裡的行話（VIP 客戶、黑名單…）每個都寫
  「系統要怎麼認出它」（例：VIP 客戶＝會員資料的等級欄位是 VIP）。
  數字類的詞也算（總金額、可用額度…）：怎麼算、含不含稅或折扣，要定義。
- **會用到哪些其他系統、廠商或資料**（選填）：Email、簡訊、簽核系統、外部廠商…列出來，
  要查的資料在哪也寫——開發才知道要去接誰。
- **輸入**（每個參數一列的表）：**名稱、型態、必填？、預設值、輸入限制/要先檢查什麼、說明**。
  - 參數＝使用者要提供給這個功能的資料。型態＝這格是哪一種資料，用你會的話寫：
    文字／整數／小數／是-否／日期時間／清單。
  - 「輸入限制」寫要不要先檢查值：不能是空的？要是正數？有固定格式（像 Email）？
  - **單號/憑證類**欄位（簽核單號、核准文號…）多寫三件事：去哪裡查、查到什麼才算有效
    （存在就好？要已核准？要對得上這筆資料？）、同一個號碼能不能重複用。
  - **某情況才必填**的，把那個情況寫清楚；格式只是舉例就註明「舉例，開發定」。
  - 金額類參數寫幣別跟小數位（例：新台幣、整數）。
  - 參數名稱寫不出英文就寫中文，開發會取名。
- **輸出**（每個欄位一列的表）：**名稱、型態、說明**。查詢類＝會查到的欄位**全部列出來**；
  變更類＝做完會回什麼（過程拿到的重要編號要不要回也寫）。回清單的話寫**照什麼順序排**、
  大概多少筆、全回還是回一部分。結果馬上生效還是有延遲，寫清楚。
- **開始前先確認**（選填）：動手前業務上要先確認什麼，有幾條列幾條。
  條件式規則**兩邊都寫**：什麼情況要、什麼情況不用。
- **步驟**（依序，**寫詳細**）：整份 SOP 的核心，越詳細 AI 越不用猜。
  人工怎麼做就怎麼寫；一步寫不清楚就拆兩步。每步照執行順序填三格：
  **做之前先確認（選填）→ 怎麼做（必填）→ 做完怎麼確認（選填）**。
  信件/畫面要顯示的資訊（像「預計完成時間」），**內容從哪來**也要講。
  處理對象有不同類型、行為不一樣的（個人帳號 vs 公司帳號），每種都寫清楚。
- **全部做完，怎麼算成功**（選填）：例「帳號查不到了、使用者有收到通知」。
- **做了之後能復原嗎**：能→寫怎麼復原；不能→明寫「做了就回不去」，
  並寫做之前要不要額外把關、怎麼把關。
- **同一個對象能重複做嗎**：做過再做會怎樣（擋？允許？有上限？）；
  做過一部分之後再做的規則。
- **可能出什麼錯、出錯了怎麼辦**：用平常的話寫。「開始前先確認」的**每一條**都要有
  對應的一條「沒過怎麼回應」。**主要動作之後的收尾步驟**（標記單據、寄信…）失敗
  也要各寫一條——別只寫主要動作的失敗。有碰外部系統的，
  **「對方沒回應、不知道成功沒有」怎麼辦**一定要寫（等人工確認？自動再試？先鎖住？）
  ——這條最容易漏、也最貴。要人工接手的，寫清楚誰接、怎麼接。
- **測試例子**（想得到的情況**全部列**）：「**這樣輸入 → 應該得到這樣的結果**」，
  輸入用「輸入」表的參數寫。覆蓋規則：成功至少一個；「開始前先確認」與「可能出什麼錯」
  的每一條都要有對應例子；特殊情況（空值、重複做、做一半）也列。
  這些例子會直接變成 AI 自我驗證的依據，**很重要，別跳過**。
- **要留什麼紀錄、給誰看**：記什麼、給誰用；**失敗/被擋的要不要也記一筆**，寫清楚。

## 你填的每一格，會變成規格的什麼

（你不用懂右邊，只是讓你知道每格都有用、不是白填的。）

| 你填的 | 會變成 | 讓 AI 能 |
|--------|--------|---------|
| 做什麼（詳細說明） | 功能的用途與範圍 | 知道在做哪件事、為誰做 |
| 誰可以用 | 權限控管 | 擋掉沒權限的人 |
| 特別名詞表 | 分類判斷規則 | 認得出你的行話、條件判斷不出錯 |
| 其他系統表 | 外部介接清單 | 知道要接誰、資料去哪查 |
| 輸入表（名稱/型態/預設值/限制） | 參數定義與格式檢查 | 收對資料、擋錯資料 |
| 輸出表（名稱/型態） | 回傳格式 | 回對資料 |
| 開始前先確認 | 前置條件檢查 | 擋掉不該做的操作 |
| 步驟（含每步前後確認） | 執行流程與中途失敗的處理 | 照對的順序做、卡住知道算什麼 |
| 全部做完怎麼算成功 | 成功情境的驗收 | 知道對的樣子 |
| 能復原嗎/能重複做嗎 | 風險等級與重複執行規則 | 危險操作多一道把關、重複請求照你的規則 |
| 會出什麼錯 | 失敗情境的驗收與錯誤表 | 知道錯的樣子、怎麼回應 |
| 測試例子 | 驗收準則與測試計畫 | 自己驗證有沒有做對 |
| 要留什麼紀錄 | 操作紀錄（日後查帳/查問題用） | 每個操作留下可查的紀錄 |

## 填不出來怎麼辦

- **業務判斷填不出來**（例：這動作到底能不能反悔、失敗該怎麼處理）→ 這正是只有你知道的事，
  問清楚業務再填，別留空。
- **不確定是不是真的規則** → 別當事實寫，句尾標「（假設，待確認）」——你隨手猜的規則，
  後面會被全部當真做出來，事後才發現要拆掉最貴。
- **參數細節寫不精確**（例：型態不知道怎麼講、名稱想不出英文、格式要問別的部門）→
  用白話寫、用中文取名，開發看得懂就好；真的不知道的格子標「請開發補」或「待確認：問○○」，
  別整格空著。
- **軟體自己會處理的事**（例：登入怎麼驗、要不要防同時操作）→ **不用管**，
  那是 skill 跟技術審批者的事。
