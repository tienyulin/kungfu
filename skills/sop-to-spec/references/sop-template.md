# <這份 SOP 做什麼：一句話>
> 提示：例「把使用者帳號的管理（查詢與刪除）變成 API」。

## 開工前整體確認（選填，有就依序列）
> 提示：整份 SOP 的功能開始用之前，整體要先成立什麼？沒有就刪掉這節。例「操作者要有管理員權限」。

## 功能們（每隻 API 一塊）
> 提示：只放**同一組**的功能（都在動同一個東西，例如「帳號」的新增/查詢/修改/刪除），
> 一隻 API 一塊，幾隻就幾塊。不同組的（訂單、通知…）另開一份 SOP，不要擠在這份。

### 功能 1：<做什麼>
> 提示：例「刪除一個使用者帳號」。
- 輸入（要提供什麼資料）：
  > 提示：例「使用者 ID」。
- 輸出（做完回傳/得到什麼；查詢類＝會查到哪些欄位）：
  > 提示：例「回傳被刪帳號的 ID 與刪除時間」。
- 開始前先確認（這隻功能的 precheck，選填，有幾條就依序列幾條）：
  > 提示：整隻功能動手前，業務上要先確認什麼。例「先確認這個帳號沒有未結清的訂單」。
- 步驟（依序；簡單功能寫一步就好）：
  > 提示：這隻功能實際上怎麼做，照順序列。每一步可以帶自己的「做之前確認」「做完確認」，沒有就不寫。
  1. <這步做什麼>
     - 做之前先確認（選填）：
     - 做完怎麼確認（選填）：
  2. <這步做什麼>（照上面複製，幾步就幾條）
  > 提示：例「1. 停用帳號（做完確認：帳號不能再登入） 2. 通知使用者 3. 刪除帳號資料（做完確認：查不到了）」。
- 全部步驟做完，怎麼算這隻功能成功（postcheck，選填）：
  > 提示：例「帳號查不到了、狀態變已刪除、使用者有收到通知」。
- 危險嗎/能反悔嗎/要誰批准：
  > 提示：會不會無法復原、要不要人簽核。不能反悔的一定要寫要誰批准、憑什麼批准（例：變更單號）。
- 可能出什麼錯、現場怎麼處理（含哪一步出錯怎麼辦）：
  > 提示：用平常的話。例「查無此帳號→回報找不到；帳號還有未結訂單→擋下不刪；通知寄不出去→照樣刪，記下來補寄」。

## 全部功能一起看，整體驗什麼（選填）
> 提示：這組功能整體要成立什麼。沒有就刪掉這節。

## 怎樣算成功、怎樣算失敗
> 提示：一句話講清楚成功與失敗的界線（可分功能寫）。

## 測試例子（每隻功能至少 1 個：這樣輸入 → 該得到這樣結果）
> 提示：別跳過，這幾個例子是 AI 驗證自己有沒有做對的依據。
> 例「刪一個沒訂單的帳號 → 成功，帳號消失」「刪一個有未結訂單的帳號 → 被擋，帳號還在」。

## 要留什麼紀錄、給誰看
> 提示：每隻功能要記什麼。例「記操作人、時間、刪了哪個帳號」。
