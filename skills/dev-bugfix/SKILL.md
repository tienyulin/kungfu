# dev-bugfix

**開工前：把下面的進度清單照抄進你的回覆，每完成一步打勾再做下一步。**
遇到本 skill 沒定義的情況：停下來問使用者，不要自行發明。

```
進度：
- [ ] Step 1 重述 bug（三欄齊全）
- [ ] Step 2 重現（貼逐字失敗輸出）
- [ ] Step 3 定位（讀過可疑路徑每個函式）
- [ ] Step 4 根因一句話（無 probably/maybe/可能）
- [ ] Step 5 最小修復
- [ ] Step 6 驗證（重現指令轉綠 + 測試套件）
- [ ] Step 7 完成報告
```

## Step 1 — 重述 bug

輸出（照格式填，三欄缺一不可）：
```
BUG: given <輸入/操作>, got <實際結果>, expected <預期結果>.
```
從使用者訊息填不齊三欄 → 只問缺的那一欄，停下等回覆。

## Step 2 — 重現

找到或寫出一條能展示 bug 的指令（失敗測試、curl、執行腳本）。
輸出＝該指令＋**逐字**失敗輸出。貼不出實際輸出＝沒重現。

試 3 次仍不能重現 → 照抄此模板回報後**停**：
```
CANNOT REPRODUCE
TRIED: 1) <指令+結果> 2) <指令+結果> 3) <指令+結果>
NEED: <缺什麼 - 完整錯誤原文 / 環境 / 操作步驟>
```
禁止修一個你沒親眼看到的 bug——那是猜，不是修。

## Step 3 — 定位

從錯誤往回追到源頭。可疑路徑上每個函式都要**讀實際程式碼**；
禁止用函式名猜行為。錯誤訊息只准逐字引用，不准憑記憶改寫。

輸出：`SUSPECT: <file:line> - <為什麼可疑>`

## Step 4 — 根因

輸出：`ROOT CAUSE: <X> happens because <Y>.`

兩道自檢，任一不過就回 Step 3：
1. 句子含 probably / maybe / 可能 → 還沒查完；真查盡了就明說不確定並問使用者。
2. Symptom patch 檢查（治標不治本）：計畫的修法只是把壞值包起來（try/catch、if-null、`?.`），
   卻說不出壞值**從哪裡產生** → 不是根因。

## Step 5 — 最小修復

修在根因處，動最少行。禁止順手改名、重排格式、修無關問題
（無關問題寫進 Step 7 的 NOTES）。除錯試錯要一次一個改動，沒效先 revert 再試下一個。

## Step 6 — 驗證（機器 gate，紅了不准進 Step 7）

1. 重跑 Step 2 的重現指令 → 必須轉綠，貼輸出。
2. 跑既有測試套件（或最相關子集），貼結尾統計行。
3. 有測試框架 → 加一條 regression test（沒有此修復時會失敗的那種）。

只准回報**實際執行過**的驗證並附輸出；沒跑的寫「未跑」。
同一錯誤修 3 次不同做法仍在 → 停手，用 STUCK 模板回報（見常駐 Constitution Law 8；
未安裝 Constitution 時：列三次嘗試＋逐字錯誤＋最佳猜測＋一個具體問題）。

## Step 7 — 完成報告（照抄填空）

```
FIXED: <一句話 - 根因 + 改了什麼>
FILES: <清單>
VERIFIED: <重現指令轉綠輸出 + 測試統計行，逐字>
NOTES: <假設、順路發現的無關問題 - 沒有就寫 none>
```

## 完成定義
- [ ] 七步全打勾，順序沒跳
- [ ] Step 2 與 Step 6 都貼了逐字指令輸出
- [ ] diff 只含修復本身（無改名/重排/無關修）
