# agent-rules-refactor

**開工前：把下面的進度清單照抄進你的回覆，每完成一步打勾再做下一步。**
遇到本 skill 沒定義的情況：停下來問使用者，不要自行發明。

鐵律：**重構＝行為零改變**。行為變了就不是重構，必須明說並取得同意。

```
進度：
- [ ] Step 1 綠燈檢查（動工前測試全綠）
- [ ] Step 2 範圍一句話
- [ ] Step 3 小步變換（每步跑測試）
- [ ] Step 4 行為稽核
- [ ] Step 5 完成報告
```

## Step 1 — 綠燈檢查（動工前）

先跑測試套件，貼結尾統計行。三分流：
- **全綠** → 繼續。
- **已有失敗** → 停。回報哪些在失敗，問使用者：先修它們，還是記錄在案後照樣重構
  （免得之後被算到你頭上）？等回覆。
- **沒有測試** → 停。告知盲飛風險，給三選項等使用者挑：
  (a) 先為目標程式碼寫 characterization tests（記錄現狀行為的測試）
  (b) 小心進行，靠手動驗證
  (c) 中止

## Step 2 — 範圍一句話

```
REFACTOR: <目標檔案/模組> - <目的，例：把重複的 parsing 抽成一個函式>
```
這行以外的東西一律不碰。

## Step 3 — 小步變換

一次只做一個機械變換（extract function / rename / move / inline）。
**每步之後跑測試。** 紅了 → 立刻 revert 該步；禁止在壞掉的重構上繼續往前修。

## Step 4 — 行為稽核

回報前過一遍 diff，自問：任何改動有沒有動到可觀察行為
（輸出、拋出的錯誤、API 形狀、side-effect 順序）？
有 → 要嘛 revert，要嘛明確標成「行為改變，需你簽核」。

## Step 5 — 完成報告（照抄填空）

```
REFACTORED: <一句話>
BEHAVIOR CHANGES: none   （或逐條列出 - 每條都要使用者簽核）
TESTS: <動工前統計行> → <完工後統計行>   （兩者必須一致）
FILES: <清單>
```

## 完成定義
- [ ] Step 1 有貼動工前測試輸出
- [ ] TESTS 前後統計一致
- [ ] BEHAVIOR CHANGES 為 none，或每條已標註待簽核
