# DIAGRAMS — 流程圖/架構圖/時序圖繪製規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：畫任何技術圖 — 流程圖、架構圖、時序圖、狀態圖、ER 圖、部署圖。
> 每條規則三段：觸發（可觀察）→ 動作（可執行）→ 自檢（yes/no）。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。本檔沒寫到的情況，用 KERNEL.md。
> 核心：**圖是給讀者三秒鐘抓到重點用的，不是把知道的東西全部倒出來。**
> 畫「設計中的系統」→ 內容依 domains/ARCHITECT.md；本檔管怎麼畫。

## R1 — 先選對圖型，再動筆

- **觸發**：被要求「畫個圖」。
- **動作**：先判斷要回答的問題是哪型，照表選圖；使用者指定的圖型跟問題對不上 → 照他的畫，附一句建議。

| 要回答的問題 | 圖型 |
|---|---|
| 步驟怎麼走、在哪裡分岔 | flowchart |
| 誰先呼叫誰、訊息順序 | sequence diagram |
| 系統由什麼組成、誰依賴誰 | component / 架構圖（C4） |
| 一個東西有哪些狀態、怎麼轉換 | state diagram |
| 資料長什麼樣、表怎麼關聯 | ER diagram |
| 跑在哪些機器/環境上 | deployment diagram |

- **自檢**：我能一句話說出「這張圖回答什麼問題」嗎？
- **例**：「畫一下登入流程」——重點是前端、後端、DB 之間的往返 → sequence diagram，不是 flowchart。

## R2 — 一張圖一個問題，節點上限 15

- **觸發**：草稿的節點（框＋菱形＋參與者）超過 15 個。
- **動作**：拆。拆法照抽象層級：先畫一張「全景圖」（子系統層級，≤10 框），每個需要細節的子系統各畫一張子圖。拆分線落在子系統或階段邊界，不准從流程中間硬切。多張圖用編號＋一句話目錄串起來。
- **自檢**：每張圖數過節點 ≤15 嗎？
- **例**：錯：一張 40 框的「全系統架構圖」（誰都看不懂，等於沒畫）。→ 對：1 張 context 圖＋3 張子系統圖。

## R3 — 同一張圖只站一個高度

- **觸發**：檢查草稿的每個節點。
- **動作**：用同一個問句測全部節點，例如「這是一個服務嗎？」——必須全 yes 或全 no。混層的節點（一顆「PostgreSQL 的 orders 表」出現在服務層級圖裡）→ 收進上層框，細節移到子圖。
- **自檢**：同一問句測所有節點，答案一致嗎？
- **例**：錯：圖上同時有「支付系統」（一整個子系統）和「檢查 email 格式」（一行程式）。→ 對：兩個高度拆兩張圖。

## R4 — 箭頭一種語義，條條有標籤

- **觸發**：畫任何一條線之前。
- **動作**：1. 先宣告這張圖的箭頭是什麼意思（呼叫方向／資料流向／依賴方向——三選一，混用 = 圖壞掉）。2. 每條箭頭都有標籤：動作或協定（「HTTP POST /orders」「發布 order.created」）。3. 同步用實線、非同步用虛線，並在圖例聲明。
- **自檢**：圖上有沒有任何一條裸箭頭（無標籤）？箭頭語義是不是只有一種？
- **例**：錯：A → B 裸箭頭（讀者不知道是 A 呼叫 B 還是資料從 A 流到 B——兩種解讀方向相反）。→ 對：「A —HTTP GET→ B」。

## R5 — 畫現狀就要對照現場，不畫想像

- **觸發**：圖描述的是**已存在**的系統（code、設定、基礎設施）。
- **動作**：1. 每個節點和連線要在原始碼／設定／文件裡指得出對應物（grep 到的 route、connection string、docker-compose 服務名）。2. 對照時發現現場跟你以為的不同 → 照現場畫。3. 規劃中但還不存在的元件 → 虛線＋標「規劃中」。4. 圖型宣告在標題：「現狀」或「目標設計」，兩者禁止混在同一張不標示。
- **自檢**：現狀圖的每個節點，我都指得出它在哪個檔案/設定裡嗎？
- **例**：錯：憑印象畫了一個「cache 層」（實際上兩年前就拆掉了），新人照圖查了一下午。→ 對：docker-compose 裡沒有 → 不畫。

## R6 — 決策點兩條腿，外部呼叫要有失敗路徑

- **觸發**：flowchart 裡出現菱形（決策點），或 sequence 圖裡出現跨系統呼叫。
- **動作**：1. 每個菱形兩條出邊都畫且都標（yes/no、成功/失敗）——只畫一條 = 另一半流程是黑洞。2. 每個外部呼叫（第三方 API、DB、另一服務）至少標出失敗/timeout 走向：重試？回錯誤？進死信？3. 失敗路徑讓圖爆掉 15 節點上限 → 主圖畫 happy path＋一句「失敗處理見圖 N」，另張圖畫全。
- **自檢**：有沒有只有一條出邊的菱形？有沒有沒交代失敗的外部呼叫？
- **例**：錯：「呼叫金流 API」直接接「顯示成功頁」。→ 對：加 alt 塊：成功／逾時（重試 1 次）／拒絕（顯示失敗＋記錄）。

## R7 — 文字格式優先（Mermaid / PlantUML / D2）

- **觸發**：要產出圖檔。
- **動作**：1. 預設用文字格式（Mermaid 優先——GitHub/GitLab 原生渲染），存進 repo，跟著版本控制走。2. 使用者明說要圖片才輸出圖片，且同時附上文字源檔。3. 選語法前先確認目標平台支援（Mermaid 版本差異大，用到的圖型先查該平台的版本支援）。
- **自檢**：交付物裡有可版控的文字源檔嗎？
- **例**：文字格式的圖，code review 時 diff 一行就看到改了哪條線；PNG 的 diff 是兩坨二進位。

## R8 — Render 過才算畫完

- **觸發**：圖的源碼寫完，準備交付。
- **動作**：1. 實際 render 一次（mermaid-cli、kroki、或平台 preview），看到成品。2. Render 完檢查兩件事：語法過了、**視覺可讀**（沒有文字溢出、沒有大量交叉線）。3. 環境沒有 renderer → 逐行對照官方語法文件檢查，交付時標 `NOT VERIFIED - 未 render，請在 <平台> 預覽確認`。
- **自檢**：我看過 render 後的成品，或標了 NOT VERIFIED 嗎？
- **例**：錯：「圖大概長這樣」貼一段沒跑過的 Mermaid（第 3 行少個引號，使用者貼上去一片空白）。→ 對：render 過的輸出＋源碼一起交。

## R9 — 標題、範圍、圖例，一個都不能少

- **觸發**：任何一張要交付的圖。
- **動作**：1. 標題寫清楚視角與範圍：「訂單服務元件圖（現狀，2026-07）」，不是「架構圖」。2. 用了超過一種形狀／顏色／線型 → 必附圖例；顏色只准編碼資訊（同步/非同步、既有/新增），不准當裝飾。3. 圖上寫日期——圖是會過期的快照。
- **自檢**：蓋住其他內容只看標題，讀者知道這張圖在講哪塊、哪個時間點嗎？
- **例**：錯：三種顏色的框，顏色只是「好看」（讀者花十分鐘找不存在的規律）。→ 對：圖例「藍=既有、綠=本次新增」。

## R10 — 閱讀方向一致，交叉線是重畫訊號

- **觸發**：render 後檢視版面。
- **動作**：1. 流向只選一個方向（上到下或左到右），主流程不准中途折返。2. 交叉線超過 3 處 → 不是加彎就能救，是分組錯了：把互動密的節點挪近、或按 R2 拆圖。3. 主路徑放最顯眼的直線上，例外和旁支往兩側掛。
- **自檢**：主流程能用手指一筆畫到底嗎？交叉 ≤3？
- **例**：錯：箭頭上下左右亂竄的「義大利麵圖」。→ 對：主流程一直線往下，錯誤處理統一掛右側。

## R11 — 圖配三行字：看什麼、重點是什麼

- **觸發**：把圖放進文件或回覆。
- **動作**：圖下方配 2–4 行說明：1. 這張圖回答什麼問題。2. 讀者該注意的一個重點（瓶頸、單點、關鍵路徑）。3. 有簡化的話，簡化了什麼（「略去監控與 log 路徑」）。圖不能自己解釋自己的取捨。
- **自檢**：圖下面有說明文字，且點出一個具體重點嗎？
- **例**：錯：光禿禿一張圖丟在文件裡。→ 對：圖＋「注意：所有寫入都經過單一 primary，這是目前的吞吐上限。」

## 回合終檢（結束前逐題答，任一 no → 先修再結束）

1. 圖型對得上要回答的問題？
2. 每張 ≤15 節點、同一高度、箭頭一種語義且條條有標籤？
3. 現狀圖的每個節點都對照過現場？
4. 決策點兩條腿、外部呼叫有失敗路徑？
5. Render 過（或標了 NOT VERIFIED）？
6. 標題含範圍與日期、有圖例、圖下有三行說明？

## 禁語表

| 禁語 | 替代寫法 |
|---|---|
| 「圖大概長這樣」 | Render 過再交，或標 `NOT VERIFIED - 未 render` |
| 「細節都在圖裡了」 | 圖＋三行字：回答什麼問題、注意哪個重點、簡化了什麼 |
| 「顏色讓圖好看一點」 | 顏色只編碼資訊，進圖例；不然全部同色 |
| 「全部畫在一張比較完整」 | 超過 15 節點就拆：一張全景＋N 張子圖 |
| 「箭頭方向差不多啦」 | 宣告箭頭語義（呼叫/資料/依賴），一張圖一種 |
