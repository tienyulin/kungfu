# CODING — 寫程式規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：在專案中新增或修改程式碼（修 bug 另見 domains/DEBUG.md）。
> 每條規則三段：觸發（可觀察）→ 動作（可執行）→ 自檢（yes/no）。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。本檔沒寫到的情況，用 KERNEL.md。
> 規則依防災重要性排序，衝突時取編號小的。

## R1 動手前先讀鄰居

- **觸發**：要在既有專案裡新增或修改程式碼。
- **動作**：1. 把要改的檔案**全文**讀一遍。2. 用關鍵字搜尋專案，確認要寫的功能是否已存在（搜功能名的同義詞至少 2 個）。3. 找一個同類型的既有檔案當風格範本（命名、註解密度、錯誤處理方式照它）。
- **自檢**：我讀過目標檔案全文，且搜過重複功能嗎？
- **例**：要寫日期格式化 → 錯：直接寫一個新的 `formatDate`。→ 對：搜 `format` `date`，發現 `utils/date.ts` 已有，直接用。

## R2 沒讀過定義的 API 不准呼叫

- **觸發**：要呼叫一個本回合沒看過定義的函式、方法、端點、library API。
- **動作**：1. 打開定義（原始碼）或官方文件，讀 signature 與參數說明。2. 是第三方套件 → 先查專案實際安裝的版本（lockfile / requirements），看**該版本**的文件。3. 找不到定義 → 明說「找不到 X」，禁止假設它存在。
- **自檢**：這次 diff 呼叫的每個 API，我本回合都看過定義嗎？
- **例**：錯：憑記憶寫 `client.messages.create(model=..., max_length=...)`（參數名是編的）。→ 對：開文件，正確參數是 `max_tokens`。

## R3 Diff 只做被要求的事

- **觸發**：檢視自己的 diff，發現有請求以外的改動（改名、重排格式、重構、「順手修」）；或 diff 超過 3 個檔案 / 約 150 行而使用者沒要求那麼大。
- **動作**：1. 超標 → 停，重讀原始請求，確認範圍。2. 請求外的改動全部移除。3. 想做的事寫成一句建議放報告尾。
- **自檢**：diff 的每一行都指得出對應的使用者要求嗎？
- **例**：被要求加一個欄位 → 錯：diff 裡還重排了整個檔案的 import 順序（review 時真改動被淹沒）。→ 對：只有欄位相關的行。

## R4 跑過才交

- **觸發**：準備交付任何程式碼。
- **動作**：1. 實際執行（跑程式、或至少編譯 / type check / lint）。2. 跑相關測試。3. 把指令和輸出逐字貼進回報。4. 環境跑不了 → 回報寫「NOT RUN: <原因>」＋「TO VERIFY: <指令>」。
- **自檢**：回報裡有本回合的執行輸出，或明確的 NOT RUN 標記嗎？
- **例**：錯：「寫好了，應該能動。」→ 對：「`npm test` 通過（輸出如下）」或「本環境無 Node，NOT RUN；請跑 `npm test`」。

## R5 新邏輯配一個會壞的檢查

- **觸發**：寫了包含分支、迴圈、解析、金額計算、權限判斷的新邏輯。
- **動作**：留下一個最小可跑檢查：一個 test case，或一段 assert 自檢。標準：**這段邏輯寫壞時，該檢查必須會失敗**。一行的 trivial 改動不需要。
- **自檢**：故意把邏輯改錯一個字，這個檢查會紅嗎？
- **例**：寫了折扣計算 → 錯：沒測試，「看起來對」。→ 對：一個 test：輸入 100 元＋八折 → 斷言 80；輸入 0 元 → 斷言 0。

## R6 外部輸入過三關

- **觸發**：函式接收來自程式外部的輸入（使用者輸入、檔案、網路、命令列參數）。
- **動作**：對每個輸入寫下三種情況的行為：1. 空／缺失。2. 超大／超長。3. 格式非法。行為只能二選一：正確處理，或立刻報出可讀錯誤。禁止未定義行為。
- **自檢**：三種輸入我都寫了對應處理或拒絕嗎？
- **例**：讀 CSV 的函式 → 錯：只處理格式完美的檔案，空檔案時噴 IndexError。→ 對：空檔回空列表；缺欄位報「第 N 行缺欄位 X」。

## R7 錯誤不准吞

- **觸發**：你寫出了 `except: pass`、空的 `catch {}`、或忽略回傳的錯誤值。
- **動作**：改成二選一：1. 能處理 → 處理，並 log 原始錯誤。2. 不能處理 → 往上拋。「先吞著讓它不要炸」不是選項——那會把今天的錯誤變成三週後查不出的靈異現象。
- **自檢**：每個 catch 區塊裡都有 log 或 re-raise 嗎？
- **例**：錯：`try { sync() } catch (e) {}`。→ 對：`catch (e) { logger.error("sync failed", e); throw e; }`。

## R8 秘密與環境值不進程式碼

- **觸發**：程式碼中出現 API key、密碼、token 的實值，或絕對路徑、寫死的 URL / port。
- **動作**：1. 秘密 → 改讀環境變數，並確認該值沒進 commit 歷史。2. 環境值 → 移到設定檔或檔案頂部常數區。3. 已經 commit 過秘密 → 立刻回報使用者（該秘密視同已洩漏，需作廢重發）。
- **自檢**：在 diff 裡搜 `key`、`password`、`token`、`secret` — 有任何實值嗎？（有 = 違規）

## R9 新依賴先提案

- **觸發**：想 import 一個專案目前沒有的套件。
- **動作**：1. 先查 stdlib 或既有依賴能不能做到 → 能就用。2. 不能 → 問使用者，格式：「需要 X 功能。不加套件要自己寫約 N 行；加 <套件名>（維護狀態、大小）一行搞定。加嗎？」
- **自檢**：這次 diff 有沒有未經同意的新依賴？
- **例**：要 deep copy → 錯：裝 lodash。→ 對：stdlib `structuredClone` 就有。

## R10 一次性抽象直接刪

- **觸發**：你寫了 interface / base class / factory / config 選項，而它目前只有一個實作、一個呼叫者、或一個永不變的值。
- **動作**：刪掉抽象層，直接寫具體版本。「以後可能用得到」不是理由——以後的需求長什麼樣沒人知道，到時再抽。
- **自檢**：每個抽象層都有 ≥2 個現存的實際使用者嗎？
- **例**：錯：為一個 email 通知寫 `NotificationProvider` interface＋factory。→ 對：一個 `sendEmail()` 函式。

## R11 改既有函式先看呼叫者

- **觸發**：要修改既有函式的參數、回傳值、或行為。
- **動作**：1. 搜尋該函式所有呼叫點，列成清單。2. 逐一確認：改動後它還正確嗎？3. 不相容的呼叫點一併修，或改用新增函式而不動舊的。
- **自檢**：呼叫點清單我列過且逐一看過嗎？
- **例**：把回傳值從 `null` 改成拋例外 → 錯：只改函式本身，三個呼叫點的 `if (result === null)` 全部靜默失效。→ 對：grep 出 3 個呼叫點一併改。

## R12 註解只寫程式碼看不出的事

- **觸發**：要寫註解。
- **動作**：只寫約束和原因（「為什麼是 3 次：上游 API 限流」）。刪掉：翻譯型註解（「// 迴圈陣列」）、給 reviewer 的話（「// 這裡我改成了…」）、註解掉的舊碼（版控有，刪）。
- **自檢**：每條註解都在說「程式碼本身讀不出來的事」嗎？

## 回合終檢（結束前逐題答，任一 no → 先修再結束）

1. 讀過目標檔案全文＋搜過重複功能？
2. 每個呼叫的 API 都看過定義？
3. 跑過並貼了輸出（或標了 NOT RUN）？
4. diff 每行都對應使用者要求？
5. 沒有吞掉的錯誤、沒有 hardcode 的秘密？
6. 新邏輯有一個「改壞會紅」的檢查？

## 禁語表

| 禁語 | 替代寫法 |
|---|---|
| 「這樣寫應該能動」 | 跑它。「`<指令>` 通過（輸出如下）」或「NOT RUN: <原因>」 |
| 「我假設這個函式存在 / 參數是這樣」 | 打開定義看。找不到就寫「找不到 X 的定義」 |
| 「順便重構 / 清理了一下」 | 不准順便。「另外注意到 X，需要的話我可以處理」 |
| 「以後可能用得到，先加了」 | 刪掉。需求出現時再加 |
| 「測試之後再補」 | 現在補，或回報列 NOT DONE |
