# EDGE-CASES — Edge Case 挖掘規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：替任何東西找 edge case — 程式、API、計畫、流程、產品設計、合約條款。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。
> 核心：edge case 不是靠靈感想出來的，是**拿生成器表對著每個輸入機械式輾過去**輾出來的。
> 靈感會漏，表不會。

## 方法（五步，依序）

### 第 1 步 — 先列輸入清單，edge case 藏在你沒列出來的輸入裡

- **動作**：列出對象的**全部**輸入，包含隱形的：參數、檔案、環境變數、設定、**時間**（現在幾點也是輸入）、**既有狀態**（DB 裡已有什麼）、使用者身分/權限、上游依賴的回應。
- **自檢**：清單裡有至少一個非顯性輸入（時間/狀態/環境）嗎？沒有 = 還沒列完。
- **例**：「檢查折扣函式」→ 顯性輸入：訂單、折扣碼。隱形輸入：當下時間（活動期限）、使用者歷史（首購限定）、幣別設定——三個隱形的才是事故高發區。

### 第 2 步 — 每個輸入 × 生成器表，逐格問「這格在這裡是什麼」

拿下方五張生成器表，對清單上**每一個**輸入逐格套。每格問：「這個生成器套在這個輸入上，具體是什麼情境？」寫得出來就記下，明顯不適用就跳過。禁止「這個輸入很單純」整個跳過——單純是靈感的說法，表不接受。

### 第 3 步 — 規格反查：每個「必須／總是／絕不」都構造一個違反情境

- **動作**：掃對象的規格/註解/文件，每出現一個絕對詞（必須、一定、總是、絕不、唯一），就問：「我怎麼讓它不成立？」構造那個情境。絕對詞是作者的假設清單，也就是破口清單。
- **例**：「每個訂單總是有一個使用者」→ 使用者被刪除後他的舊訂單呢？匿名結帳呢？

### 第 4 步 — 對抗者視角：想佔便宜的人會輸入什麼

- **動作**：換三頂帽子各問一次：**貪婪者**（怎麼利用規則佔便宜：負數量退錢、折扣碼疊加、重放請求）、**破壞者**（怎麼弄壞它：注入字元、超大 payload、亂序呼叫）、**笨拙者**（不看說明的人會怎麼用：按兩次送出、中途按返回、複製貼上帶隱形字元）。
- **自檢**：三頂帽子各產出至少一個情境了嗎？

### 第 5 步 — 排序與處置：不是每個 edge case 都要修

- **動作**：找到的每個 case 按**機率 × 傷害**排序，分三類處置：1. **處理**（寫邏輯正確支援）。2. **擋掉**（明確拒絕＋可讀錯誤——比默默算錯好一百倍）。3. **記錄接受**（「已知不處理，因為 ___」寫下來給使用者簽收）。三類都沒有 = 未定義行為 = 隱藏炸彈。
- **自檢**：每個找到的 case 都有三類之一的標籤？
- **輸出格式**：每個 case 寫成**可執行的具體情境**（「輸入 -1 件商品＋滿千折百券」），不是「要注意負數」。

## 生成器表（核心資產，逐格輾）

### 表 A — 單值輸入

| 生成器 | 套用問法 |
|---|---|
| 空 | null／空字串／欄位不存在，三種是三個不同的 case |
| 零與負 | 0、-1、-0 各是什麼行為 |
| 極大極小 | 型別上限、上限+1、超長字串、巨大檔案 |
| 邊界三點 | 每個宣告的限制 N，測 N-1、N、N+1 |
| 精度 | 0.1+0.2、四捨五入、貨幣分位、NaN／Infinity |
| 特殊字元 | 引號、換行、`<script>`、SQL 字元、emoji、全形、RTL、零寬字元 |
| 型別邊緣 | "123" vs 123、大小寫、前後空白、+886 vs 0 開頭 |
| 合法但荒謬 | 格式全對、語義離譜：生日 1800 年、價格 0 元、年齡 200 |

### 表 B — 集合輸入

| 生成器 | 套用問法 |
|---|---|
| 空集合／單元素 | 迴圈、平均值、max 在 0 個和 1 個元素時 |
| 巨量 | 10 萬個元素時的效能與分頁 |
| 重複 | 重複 key、重複 ID、全部相同值 |
| 順序 | 亂序輸入、依賴排序的邏輯遇到相同排序鍵 |
| 首尾 | 第一個和最後一個元素有沒有被特殊對待（差一錯誤） |

### 表 C — 時間

| 生成器 | 套用問法 |
|---|---|
| 時區 | 使用者、伺服器、DB 三者時區不同時 |
| 邊界日 | 2/29、月底 31 號、跨年瞬間、DST 切換 |
| 同時 | 同一毫秒兩筆、timestamp 相同怎麼排序 |
| 時鐘異常 | 時鐘回撥、client 時間造假 |
| 過期邊緣 | 剛好在到期那一秒、已過期資料被讀到 |

### 表 D — 狀態與時序

| 生成器 | 套用問法 |
|---|---|
| 重複執行 | 同一操作跑兩次（重送、重試）結果一樣嗎（冪等） |
| 中斷 | 做到一半斷掉（斷網、當機）→ 重來會怎樣、半成品狀態誰清 |
| 亂序 | 事件 B 比 A 先到 |
| 生命週期邊緣 | 剛建立就刪除、刪除後又收到它的訊息、修改進行中被讀取 |
| 併發 | 兩人同時改同一筆（read-modify-write）、同帳號兩裝置 |
| 狀態極端 | 全新使用者（什麼都沒有）vs 十年老戶（什麼都巨量） |

### 表 E — 環境與依賴

| 生成器 | 套用問法 |
|---|---|
| 依賴故障 | 上游掛／慢 10 倍／回錯資料（接 domains/CASCADE.md） |
| 資源耗盡 | 磁碟滿、記憶體滿、連線池滿、配額用完 |
| 權限 | 權限不足、權限中途被撤、跨租戶存取 |
| 中途斷網 | 請求送出後斷線——對方到底收到沒（不確定性） |
| 版本與地區 | 新舊版本並存、locale／幣別／字元集不同 |

### 非程式對象的對應（計畫、流程、合約同樣用表）

把「輸入」換成：參與者、資源、時程、外部條件。生成器不變——計畫的「空值」是關鍵人請假；流程的「併發」是兩件事撞同一天；合約的「邊界三點」是金額剛好在門檻上；活動企劃的「對抗者」是有人鑽規則薅羊毛。

## 回合終檢（結束前逐題答，任一 no → 先修再結束）

1. 輸入清單含隱形輸入（時間／狀態／環境）？
2. 每個輸入都輾過五張表（不是挑順眼的輾）？
3. 每個絕對詞都構造了違反情境？
4. 三頂對抗帽子各有產出？
5. 每個 case 有處置標籤（處理／擋掉／記錄接受）？
6. 每個 case 是可執行的具體情境，不是「要注意 X」？

## 禁語表

| 禁語 | 替代寫法 |
|---|---|
| 「要注意空值的情況」 | 「輸入為 null／空字串／欄位缺失時，預期行為各是 ___」 |
| 「這個輸入很單純」 | 單純也輾表；表跑完寫「表 A-H 均不適用」才算過 |
| 「大概涵蓋了主要情況」 | 「輸入 N 個 × 五表輾完，找到 M 個 case，清單如下」 |
| 「這種情況不太可能發生」 | 「機率低×傷害 ___ → 分類：記錄接受，理由 ___」 |
