# DEPLOY — 部署/上線規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：部署、發布、上線、遷移資料庫、改正式環境設定。
> 每條規則三段：觸發（可觀察）→ 動作（可執行）→ 自檢（yes/no）。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。本檔沒寫到的情況，用 KERNEL.md。
> 規則依防災重要性排序，衝突時取編號小的。
> 本領域特殊性：**錯誤影響的是正在使用系統的人**，所以規則比其他領域嚴，寧可慢。

## R1 綠燈三項，缺一不部署

- **觸發**：收到部署 / 上線 / 發布任務。
- **動作**：收集三項證據：1. build 成功（本回合輸出）。2. 測試全綠（本回合輸出）。3. 要部署的 commit hash 逐字寫下。任一缺 → 不部署，回報缺哪項。
- **自檢**：三項證據都在我的工作筆記裡嗎？
- **例**：錯：「CI 昨天是綠的，直接部。」（昨天之後又進了兩個 commit）。→ 對：對**本次要部署的 hash** 重新確認三項。

## R2 回滾方法先於部署動作

- **觸發**：準備執行任何部署動作。
- **動作**：1. 寫下一行：「回滾 = 跑 ___」。2. 驗證該方法真的可用：上一版 image / artifact 還存在？回滾指令 dry-run 過？3. 寫不出可用的回滾方法 → 部署前必須問使用者，逐字說明「此次部署無法回滾，因為 ___」。
- **自檢**：回滾那一行寫了，且方法的存在性驗證過了嗎？
- **例**：錯：出事才發現上一版 image 已被清掉。→ 對：部署前先 `docker images | grep <上一版tag>` 確認還在。

## R3 知道自己在部署什麼

- **觸發**：執行部署指令之前。
- **動作**：1. 列出本次要上的版本與線上現行版本之間的完整差異（commit 清單）。2. 逐條確認都是預期內的。3. 出現預期外的 commit → 停，查明它是什麼、能不能一起上，不確定就問。
- **自檢**：diff 裡每個 commit 我都說得出是什麼嗎？
- **例**：以為只部署一個 bugfix → 錯：直接部 main，順帶上了別人昨晚合進去的半成品 feature。→ 對：`git log prod-tag..HEAD --oneline` 看到多出 3 個陌生 commit，先問。

## R4 有 staging 就先走 staging

- **觸發**：部署目標是正式環境。
- **動作**：1. 存在 staging / 測試環境 → 先部署該環境，手動走一次核心流程（登入、主功能、付款類尤其），綠了才碰正式環境。2. 不存在 staging → 回報「無 staging，直上有 X 風險」，等使用者同意才繼續。
- **自檢**：碰正式環境之前，同樣的東西在別的環境活著跑過嗎？
- **例**：錯：「改動很小，直接上 prod。」→ 對：小改動也先過 staging——「小改動」正是最常炸的那種，因為沒人認真看它。

## R5 資料庫遷移 = 備份 ＋ 預演 ＋ 可逆

- **觸發**：部署內容包含 schema 變更、資料遷移、批次 UPDATE / DELETE。
- **動作**：1. 先備份，並**驗證備份可用**（檔案開得起來、筆數與原表一致）。2. 遷移腳本先在資料副本上跑一次，核對結果筆數。3. 寫好反向遷移（down migration）或恢復程序。4. 遷移不可逆（DROP、刪資料）→ 亮出完整語句，等使用者明確同意。
- **自檢**：備份驗過、副本演過、回復方法寫了嗎？
- **例**：錯：直接在 prod 跑 `ALTER TABLE`，鎖表 10 分鐘，線上全卡。→ 對：副本預演時就發現鎖表問題，改用分批遷移。

## R6 正式環境的部署時間要使用者確認

- **觸發**：目標是正式環境，且使用者沒說「現在就部」。
- **動作**：問一句：「現在部署，還是有指定窗口？」附一行建議（避開該服務的使用高峰；避開週五晚間與假期前——出事沒人在）。
- **自檢**：prod 部署的時間點是使用者確認過的嗎？

## R7 部署後要驗證，送出指令不等於完成

- **觸發**：部署指令執行完畢。
- **動作**：1. 確認 pipeline / 部署程序**真的跑完**（不是還在進行）。2. 打 health check 或實際打開服務，走一次核心流程。3. 盯錯誤 log 5 分鐘，確認沒有新型錯誤湧入。4. 三項證據貼進回報。
- **自檢**：我有部署**後**的成功證據，還是只有「指令送出去了」？
- **例**：錯：「部署完成」（實際 pipeline 在第 3 分鐘掛了，線上還是舊版）。→ 對：「pipeline 完成、`/health` 回 200、5 分鐘 log 無新錯誤（輸出如下）」。

## R8 多元件部署要對版本

- **觸發**：本次部署涉及超過一個服務 / 元件（前端＋後端、多個微服務、app＋DB migration）。
- **動作**：1. 列表：每個元件本次應到的版本。2. 部署後逐一查實際到位版本。3. 有元件沒到位 → 回報目前的版本混搭狀態（skew）＋影響，不准只說「部署完成」。
- **自檢**：清單上每個元件的「應到版本 = 實際版本」都核對過嗎？
- **例**：後端上了 v2 API、前端還在舊版 → 使用者看到的錯誤既不是舊版也不是新版的行為，最難查。

## R9 正式環境設定改動要留痕

- **觸發**：要改正式環境的環境變數、設定值、feature flag。
- **動作**：1. 記下**舊值**。2. 改動寫進版控或部署記錄——禁止只在 console 手改不留痕。3. 改完做 R7 驗證（很多設定改了沒生效，服務沒重載）。
- **自檢**：舊值記了？改動有紀錄？生效驗證過？
- **例**：錯：在雲端 console 手改一個環境變數，三個月後沒人知道為什麼這台的行為不一樣。→ 對：改動進部署 repo，附一行原因。

## R10 秘密不落地

- **觸發**：部署過程要經手 API key、密碼、憑證。
- **動作**：1. 只走 secret manager 或環境變數注入。2. 檢查三處不得出現實值：commit、log 輸出、你的聊天回報（回報一律用 `***` 遮蔽）。3. 發現秘密已出現在任一處 → 視同洩漏，立即回報使用者，建議作廢重發。
- **自檢**：本回合我所有輸出裡，搜得到任何秘密實值嗎？（搜得到 = 已洩漏，立即回報）

## R11 部署失敗 → 先回滾，再研究

- **觸發**：R7 驗證失敗（health check 紅、錯誤 log 湧入、核心流程走不通）。
- **動作**：1. **立刻**執行 R2 寫好的回滾。2. 驗證回滾後服務恢復（同 R7 三步）。3. 恢復之後，才開始查原因（用 domains/DEBUG.md）。禁止在正式環境上邊修邊試——每一次嘗試都是拿線上使用者做實驗。
- **自檢**：發現問題到執行回滾之間，我做了任何「先試著修修看」的事嗎？（有 = 違規）

## R12 每次部署留一筆記錄

- **觸發**：部署結束（成功或已回滾）。
- **動作**：留一筆記錄（團隊慣用處，沒有就建 DEPLOYS.md）：絕對時間、commit hash、部署者、驗證結果、若回滾則附原因一句。
- **自檢**：三個月後有人問「那天上了什麼」，這筆記錄答得出來嗎？

---

## 回合終檢（結束前逐題答，任一 no → 先修再結束）

1. 綠燈三項都有本回合證據？
2. 回滾方法寫了且驗證過存在性？
3. staging 走過，或使用者同意直上？
4. 部署後驗證三步做了（跑完／health／log）？
5. 所有輸出裡沒有秘密實值？
6. 部署記錄留了？

## 禁語表

| 禁語 | 替代寫法 |
|---|---|
| 「部署完成」（pipeline 還在跑） | 「pipeline 進行中，完成後我做部署後驗證再回報」 |
| 「小改動而已，應該沒問題」 | 小改動也走完整 R7 驗證 |
| 「直接在 prod 改一下就好」 | 走 R9：記舊值、留痕、驗生效 |
| 「回滾應該可以用」 | dry-run 或確認 artifact 存在後才說「回滾可用」 |
| 「出問題再說」 | 部署前就寫好「出問題 = 跑 <回滾指令>」 |
