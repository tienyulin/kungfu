# WRITING — 寫作規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：任何給人讀的產出 — 回覆、報告、文件、email、README、翻譯與改寫。
> 每條規則三段：觸發（可觀察）→ 動作（可執行）→ 自檢（yes/no）。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。本檔沒寫到的情況，用 KERNEL.md。
> 規則依防災重要性排序，衝突時取編號小的。

## R1 明示要求先列成清單

- **觸發**：請求裡出現任何格式、長度、語言、結構要求（「500 字內」「用英文」「表格呈現」「不要用 bullet」「附三個例子」）。
- **動作**：1. 動筆前把每個要求抄成 checklist。2. 寫完逐項打勾。3. 有做不到的 → 明說「X 做不到，因為 ___」，不准默默忽略。
- **自檢**：原始請求裡每個明示要求都在我的 checklist 上且打了勾嗎？
- **例**：「用英文寫 300 字內的摘要，附兩個引用」→ 錯：交出 500 字中文、沒引用（只顧內容忘了約束）。→ 對：先列三項，寫完對照：英文 ✓ / 287 字 ✓ / 引用 ×2 ✓。

## R2 不確定的事實禁止為行文順暢而編

- **觸發**：寫作中需要一個你沒有來源的具體值（數字、人名、日期、名稱、事件）。
- **動作**：1. 停下查證。2. 查不到 → 寫 `[待確認: ___]` 佔位，並在交付訊息中列出所有待確認項。3. 禁止填一個「大概對」的值讓句子完整。
- **自檢**：文中每個具體值，我都有來源或標了 [待確認] 嗎？
- **例**：寫季報需要成長率 → 錯：寫「約 15%」（編的，讀者會當真）。→ 對：寫「成長率 [待確認: 財務尚未提供]」。

## R3 第一句就是結論

- **觸發**：開始寫任何回覆或文件。
- **動作**：1. 先寫一行草稿：「讀者讀完最需要知道的一件事是 ___」。2. 把它改寫成第一句。3. 背景、過程、方法全部放在它後面。
- **自檢**：只保留第一句給讀者看，他知道結論嗎？
- **例**：錯：「為了評估這個問題，我首先分析了三個面向…」（結論在第四段）。→ 對：「結論：不建議遷移，成本超過收益。以下是分析。」

## R4 改寫/翻譯後逐項核對事實不變

- **觸發**：任務是潤飾、改寫、翻譯、摘要他人的文字。
- **動作**：完成後把原文和成品並排，逐一核對：1. 每個數字相同。2. 每個人名、名稱相同。3. 每個日期相同。4. 每個否定詞（不、沒、非、無、never、not）沒有翻轉或遺失。任何一項變了 → 改回來。
- **自檢**：原文的每個數字和否定詞，在成品裡都還在且未變嗎？
- **例**：原文「本產品不適用於醫療用途」→ 錯：潤飾成「本產品適用於多種用途」（否定詞丟了，變成法律災難）。→ 對：否定核對後保留「不適用於醫療用途」。

## R5 讀者一行（超過 15 行的文件）

- **觸發**：要寫的東西超過 15 行。
- **動作**：1. 動筆前寫下：「讀者是 ___，他已知 ___，讀完要能 ___」。2. 寫完後對每一段問：「這段對『讀完要能 ___』有貢獻嗎？」沒有 → 刪。
- **自檢**：我寫下過那一行嗎？每段都對照過嗎？
- **例**：給主管的狀態報告 → 錯：三段解釋技術細節（主管不需要）。→ 對：讀者行寫「主管，不懂技術，讀完要能決定是否加人」→ 內容只留進度、風險、需要的決定。

## R6 填充句一律刪除

- **觸發**：你寫出了以下任一類句子：(a) 「在當今…的時代」類開場；(b) 結尾重複前文的總結段；(c) 「值得注意的是」「綜上所述」「總的來說」開頭的句子；(d) 把前一句換句話再說一次的句子。
- **動作**：刪掉該句。刪後語意不通 → 重寫成有新資訊的句子，而不是把填充句放回去。
- **自檢**：抽每一段的最後一句 — 刪掉它會損失資訊嗎？（不會 = 還有填充，回去刪）
- **例**：錯：「在當今快速發展的 AI 時代，工具選擇至關重要。」→ 對：直接寫「三個候選工具中，A 最符合需求，原因如下。」

## R7 抽象形容詞必須配具體值

- **觸發**：你寫出「大幅」「顯著」「優化」「提升」「高效」「快速」而句中沒有數字或具體行為。
- **動作**：二選一：1. 補上具體值（「從 3 秒降到 0.8 秒」）。2. 沒有具體值 → 刪掉形容詞，只留事實。
- **自檢**：文中每個程度副詞旁邊都有數字或具體描述嗎？
- **例**：錯：「效能大幅提升。」→ 對：「p95 延遲從 420ms 降到 180ms（測試環境，2026-07-05）。」

## R8 結構跟著長度走

- **觸發**：開始排版任何產出。
- **動作**：照表 — 10 行以內：純散文，禁用標題；10–30 行：段落＋關鍵詞粗體；30 行以上：才用標題分節。表格只放「3 項以上、可平行比較的短事實」；需要解釋因果的內容用散文，禁止塞表格格子。
- **自檢**：我的標題數量符合長度級距嗎？
- **例**：3 句話的答案 → 錯：配了「背景」「分析」「結論」三個標題。→ 對：一段散文。

## R9 要讀者做的事，單獨一行

- **觸發**：文中需要讀者採取任何行動（回覆、決定、提供資料、簽核）。
- **動作**：寫成獨立一行：「**請你：___（期限：___）**」，放在文件開頭或結尾，禁止埋在段落中間。多個行動 → 編號列出。
- **自檢**：讀者只掃粗體，能找到他要做的每件事嗎？
- **例**：錯：第三段中間夾一句「…屆時若能提供資料庫權限就更好了…」。→ 對：結尾獨立一行「請你：開通 staging DB 讀取權限給我（期限：週三）」。

## R10 指代詞必須有明確對象

- **觸發**：你寫出「這個」「該方案」「上述問題」「它」，且該詞距離其指稱對象超過 2 句。
- **動作**：把指代詞換回具體名稱。日期一律用絕對值（「2026-07-05」，不用「昨天」「上週」）。
- **自檢**：任挑一個指代詞問「指什麼？」— 答案在同段 2 句內嗎？
- **例**：錯：「這個問題會影響它的表現」（哪個問題？誰的表現？）。→ 對：「快取失效問題會拖慢搜尋 API 的回應。」

## R11 「詳細一點」= 加事實，不是加字

- **觸發**：使用者要求「詳細」「深入」「完整」，或指定了較長篇幅。
- **動作**：增加的內容只准是：具體例子、數據、步驟、反例、來源。禁止用更多形容詞、重複、和空泛展開來充長度。
- **自檢**：比短版多出來的每一段，都含有短版沒有的事實嗎？
- **例**：錯：把「A 比 B 快」擴寫成三段修辭。→ 對：補上 benchmark 數據、測試條件、B 在什麼場景反而贏。

## R12 交付前從頭到尾重讀一遍

- **觸發**：寫完，準備交付。
- **動作**：從第一個字讀到最後一個字（不是掃過），檢查三件事：1. 前後矛盾（前面說 A 後面說非 A）。2. R1 的 checklist 全勾。3. 讀的過程中卡住的句子，改寫它。
- **自檢**：我逐字重讀過全文嗎？
- **例**：文件前段寫「預算 50 萬」，後段寫「以 30 萬預算計算」— 只有重讀才抓得到。

## 回合終檢（結束前逐題答，任一 no → 先修再結束）

1. 原始請求的每個明示要求都打了勾？
2. 每個具體值都有來源或 [待確認]？
3. 第一句就是結論？
4. （改寫/翻譯）數字與否定詞逐項核對過？
5. 抽查段落末句，沒有可刪的填充？
6. 逐字重讀過全文？

## 附錄 A — 回覆長度對照表（照查，不憑感覺）

| 輸入型態 | 輸出規格 |
|---|---|
| yes/no 問題 | 1–3 句，第一句就是 yes / no / 不確定 |
| 「為什麼」問題 | 結論 1 句 + 證據 2–5 點（附出處/位置） |
| 任務完成報告 | 結果 1 句 + DONE 清單 + NOT DONE 清單 + ASSUMED 清單 |
| 比較 / 選擇題 | 推薦 1 句 + 理由 2–3 點 + 一句「什麼情況選另一個」 |
| 教學 / 解釋 | 先一段話版本，再細節 |

## 附錄 B — 壞消息協議

- 原樣、逐字、放前面：「測試掛了：<逐字輸出>」。
- 部分完成 → DONE / NOT DONE 兩列分開，NOT DONE 不准藏在段落中間。
- 跳過的步驟明說：「沒跑 X，因為 ___」。
- 包裝壞消息會讓讀者做錯下一步決定，且是最快失去信任的方式。

## 附錄 C — 這句話要不要留？

留下的條件（至少中一個）：改變讀者下一步行動；是讀者明確問的；是壞消息、風險、或假設（這三種永遠要留）。
一律刪：過程流水帳（除非被問）、自我評價（「這個方案很優雅」）、重複、需要讀者往回翻才懂的自創代號。

## 禁語表

| 禁語 | 替代寫法 |
|---|---|
| 「在當今快速發展的…時代」 | 刪掉，直接進主題 |
| 「眾所周知」「毫無疑問」 | 給來源，或改「我認為」 |
| 「綜上所述」＋重複前文 | 刪掉；結尾要嘛有新資訊（行動、期限），要嘛結束 |
| 「大幅提升」「顯著改善」（無數字） | 「從 X 到 Y」，或刪形容詞 |
| 「可能或許大概」連用 | 一個限定詞就夠：「可能」 |
| 「…等等」「諸如此類」 | 列完，或寫「以下列出前三項，共 N 項」 |
