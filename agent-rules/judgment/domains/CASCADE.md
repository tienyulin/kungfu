# CASCADE — 失敗連鎖分析規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：兩個方向的連鎖推演 — **向後**：找到一個故障後，清點它還弄壞了什麼；
> **向前**：預測一個系統未來會怎麼連環壞。與 domains/DEBUG.md 分工：DEBUG 找根因，本檔管根因的漣漪。
> 每條規則三段：觸發（可觀察）→ 動作（可執行）→ 自檢（yes/no）。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。
> 核心：**故障很少單獨旅行。** 修好眼前的洞只是一半，另一半是它經過的路和它的同夥。

## R1 — 找到根因後，先清點污染範圍再慶祝

- **觸發**：一個 bug / 故障的根因被確認。
- **動作**：回答三題：1. **這個原因存在多久了？**（從引入的 commit / 設定變更日起算）。2. 存在期間，**經過它的每一筆資料／請求都可疑**：有沒有壞資料寫進 DB？錯誤結果進了快取？錯誤事件發給了下游？錯誤報表已經送出？3. 列**污染清單**＋各自的清理方式。修 code 只是止血，資料修復才是治療。
- **自檢**：我知道這個 bug 的「生效期間」和期間內的受害清單嗎？
- **例**：匯率算錯的 bug 修好了 → 錯：關單。→ 對：bug 上線 11 天，期間 3,400 筆訂單金額錯誤、部分已請款——清單、重算、通知，這才是任務的主體。

## R2 — 共享依賴清點：連鎖從共用的東西開始

- **觸發**：分析任何多元件系統（向前或向後皆用）。
- **動作**：列出被超過一個元件共用的每樣東西：資料庫、快取、憑證、DNS、NTP、密鑰、同一台主機、同一個 AZ、同一條網路、同一個雲帳號、同一個 rate limit。每個共享點寫一行：「它壞 → 同時倒下的有 ___」。**兩個「獨立」服務共用一顆 DB，它們就不是獨立的。**
- **自檢**：共享清單列了嗎？有沒有紙面上冗餘、實際上共用底層的假冗餘？
- **例**：主備兩套 API「互為備援」→ 共享清點發現兩套都掛在同一張 wildcard 憑證下——憑證過期日就是兩套一起死的日子。

## R3 — 連鎖推演到第三階，一階打住是最常見的失職

- **觸發**：評估「X 壞掉會怎樣」。
- **動作**：寫鏈條表，每一階都附**機制**（不是「B 也會有問題」，是「B 的 queue 開始堆積→記憶體漲→OOM」）：
  ```
  第一階：A 掛 → 誰直接受影響（呼叫者、下游、依賴者）
  第二階：每個受影響者的異常行為 → 又影響誰
  第三階：再推一層，特別注意繞回來打 A 或打共享資源的環
  ```
  推到第三階或推不動為止。**環（A→B→C→又回到 A）是最危險的形狀**，單獨圈出。
- **自檢**：每條鏈都到第三階或明寫「到此無後續」？機制都寫了？
- **例**：「認證服務掛了」→ 一階：登入失敗。二階：前端自動重試×3 → 流量變三倍。三階：重試流量打垮 API gateway → **連不需要認證的頁面也死了**——真實影響範圍在第三階才現形。

## R4 — 定時炸彈掃描：會隨時間必然爆的東西

- **觸發**：接手或審視任何系統（向前預測用）。
- **動作**：逐項掃，每項寫「爆炸日估計」：憑證/域名到期日、磁碟成長率÷剩餘空間、序號與 ID 空間（INT 上限、UUID 碰撞不算但自增要算）、token/API key 有效期、授權到期、日期邊界（月底、年底、閏日、DST）、依賴的 EOL 日。**這些不是機率性風險，是排程好的故障**——差別只在有沒有人去看日曆。
- **自檢**：每項都有日期或「不適用」？最近的爆炸日有人認領了嗎？
- **例**：掃描發現 orders 表自增 ID 是 INT，目前 19 億、月增 1 億 → 爆炸日約 2 個月後。今天改比當天改便宜一百倍。

## R5 — 負載重分配推演：一台倒下，剩下的接得住嗎

- **觸發**：系統有任何「N 台分擔」的結構（instance、replica、分片、人）。
- **動作**：算數學：目前每台負載 ___%，掛 1 台後其餘每台變 ___%——超過 100% 或超過已知的性能拐點 → 寫出雪崩鏈：「掛 1 → 其餘過載 → 最慢的再掛 → 更過載」。給出雪崩條件（幾台以下必雪崩）和答案（餘量要留多少、或掛掉時要主動降載/限流）。
- **自檢**：「掛一台」的算術做了嗎？雪崩臨界點算出來了嗎？
- **例**：3 台各跑 70% → 掛 1 台，剩 2 台各 105% → 必雪崩。「我們有 3 台冗餘」是錯覺，這個配置的正確描述是「3 台剛好，0 台冗餘」。

## R6 — 放大器清點：哪些機制會把小火吹成大火

- **觸發**：推演任何一條鏈（R3/R5）時。
- **動作**：檢查四個經典放大器，各標「正回饋（放大）／負回饋（收斂）」：1. **重試**——有 exponential backoff 嗎？有次數上限嗎？沒有 = 流量放大器。2. **health check 踢除**——慢的都被踢掉，剩下的更慢，會不會踢到全滅？3. **自動擴容**——來得及嗎？擴容本身（冷啟動、搶連線池）會不會先補一刀？4. **警報與人的反應**——警報風暴淹沒真訊號、多人同時上去改。
- **自檢**：四個放大器都查過、都標了方向？
- **例**：下游慢 200ms → 上游 timeout 重試（無 backoff）→ 下游流量瞬間 ×4 → 徹底壓死。原始問題只是「慢了 200ms」，把它變成全站故障的是重試設定。

## R7 — 恢復本身是第二波攻擊

- **觸發**：寫恢復程序、或推演「修好之後」。
- **動作**：對恢復時刻推演三件事：1. **冷快取**——重啟後 cache 全空，第一波流量直穿 DB，DB 接得住嗎？2. **積壓重放**——停機期間堆積的 queue/請求會在恢復瞬間灌下來，下游吃得下嗎？要不要限速重放？3. **同時重連**——所有 client 同一秒重連（thundering herd），連線池/握手扛得住嗎？要不要 jitter？恢復程序沒推演過這三件事 = 你會經歷第二次故障，而且它會被誤判成「還沒修好」。
- **自檢**：恢復計畫裡對這三件事各有一行對策或「不適用」嗎？
- **例**：DB 修好 → 全部服務同時重連＋queue 裡 40 萬筆積壓齊發 → DB 再倒。第二次故障不是 bug，是恢復程序設計出來的。

## R8 — 每條鏈標偵測點：走到哪一步才會有人知道

- **觸發**：R3 的每條鏈寫完後。
- **動作**：在鏈上標出「現有監控會在**第幾階**發現」。發現點在第二階之後 → 在第一階指定一個具體監控訊號（哪個 metric、什麼閾值）。**無聲的鏈最致命**——等使用者打電話進來才知道，代表整條鏈都走完了。
- **自檢**：每條鏈都有標注的偵測階？有沒有整條鏈走完都不會響的警報死角？
- **例**：憑證過期鏈的現有偵測點 = 使用者連不上（最後一階）→ 加「到期前 30 天」的檢查，把偵測點從最後一階挪到第零階。

## R9 — 鏈條排序，前三名各給一個最便宜的斷鏈點

- **觸發**：鏈條表完成，數量 >3 條。
- **動作**：1. 每條鏈打分：機率（高/中/低）× 影響（全站/部分/局部），排序。2. 前三名各找**斷鏈點**：鏈上哪個環節加什麼措施能把鏈切斷——選**最便宜**的環節（通常是加 backoff、加超時、加隔板 bulkhead、拆共享依賴其中一種），不是最貴的全面重構。3. 其餘的鏈列出來標「已知未處理」，讓使用者決定。
- **自檢**：前三名都有具體、便宜、指名環節的斷鏈措施？
- **例**：重試風暴鏈的斷鏈點不是「重寫呼叫層」，是 client 加 exponential backoff＋上限 3 次——兩行設定，鏈斷。

## R10 — 人也在鏈上

- **觸發**：任何一條鏈的恢復環節需要人動作。
- **動作**：對人環節問四題：1. 警報送到誰？那個時段他醒著嗎（值班表存在嗎）？2. runbook 存在且最近演練過嗎（沒演練過的 runbook 當它不存在）？3. 修復需要的**權限和密碼**，值班的人半夜拿得到嗎？4. 需要的第三方（雲支援、廠商）回應時間多久？人環節斷掉，技術鏈修得再好也照走完。
- **自檢**：每個人環節四題都有答案？
- **例**：恢復要進雲 console → 唯一有權限的人在飛機上——鏈條的最弱環節不是任何服務，是一個 IAM 設定。
