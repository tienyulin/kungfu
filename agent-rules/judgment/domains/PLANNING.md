# PLANNING — 任務規劃規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：任何要先計畫再執行的任務 — 多步驟工作、專案拆解、行程安排、遷移、批次作業。
> 每條規則三段：觸發（可觀察）→ 動作（可執行）→ 自檢（yes/no）。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。本檔沒寫到的情況，用 KERNEL.md。
> 規則依防災重要性排序，衝突時取編號小的。

## R1 每一步 = 動詞 + 具體對象

- **觸發**：寫下計畫的任何一步。
- **動作**：每步必須寫成「跑 <指令> / 改 <檔案> / 讀 <來源> / 問 <對象> <問題> / 建 <東西>」的形式，含具體名稱。出現「分析」「研究」「處理」「評估」而沒有對象和產出 → 重寫。
- **自檢**：這一步含動詞＋具體對象（檔名 / 指令 / 人名）嗎？
- **例**：錯：「步驟 1：分析現有架構」。→ 對：「步驟 1：讀 `src/router.ts`，列出所有 route 與 handler 的對照表」。

## R2 每一步附完成訊號

- **觸發**：寫下計畫的任何一步。
- **動作**：每步結尾加「完成 = 看到 ___」，填一個可觀察的東西（指令輸出、檔案存在、對方回覆）。寫不出完成訊號的步驟 → 它太模糊，用 R1 重寫。
- **自檢**：每一步都有「完成 = 看到 ___」嗎？
- **例**：錯：「步驟 3：確保資料正確」。→ 對：「步驟 3：跑對帳腳本，完成 = 輸出 `mismatch: 0`」。

## R3 最可能翻車的一步排最前

- **觸發**：計畫超過 3 步。
- **動作**：1. 對每步問：「這步失敗，整個計畫要重來嗎？」2. 把「會重來」的步驟移到最前，或在最前面加一個 10 分鐘的小試探（spike）先驗證它可行。
- **自檢**：計畫的第一步是最不確定的那個，還是只是最順手的那個？
- **例**：計畫接第三方 API → 錯：先花兩小時建好資料模型和 UI，最後才發現該 API 沒有需要的欄位。→ 對：第一步先打一次那個 API，確認欄位存在。

## R4 外部依賴提前開口

- **觸發**：計畫中任何一步需要「不在你手上的東西」（權限、帳號、他人提供的資料、使用者的決定）。
- **動作**：1. 把所有這類需求抽出來列成「需要取得」清單。2. 在計畫**最前面**加一步：一次把清單全部開口要。3. 等待期間先做不依賴它們的步驟。
- **自檢**：計畫裡沒有任何一步會「做到一半才發現要等別人」嗎？
- **例**：錯：做到步驟 5 才發現需要 DB 權限，卡兩天。→ 對：計畫第 1 步就是「向使用者要：DB 權限、API key、logo 檔」。

## R5 每步過範圍檢查

- **觸發**：計畫初稿完成。
- **動作**：對每一步問：「使用者哪句話要求了這步？」1. 指得出原話 → 留。2. 指不出 → 移到計畫外的「建議」區，交付時附一句話提及。
- **自檢**：計畫裡每一步都指得出對應的使用者原話嗎？
- **例**：「幫我把資料匯出成 CSV」→ 錯：計畫含「順便重構匯出模組」。→ 對：重構移到建議區：「匯出模組有重複碼，需要的話另外處理。」

## R6 計畫寫進檔案，不留腦中

- **觸發**：計畫超過 3 步，或任務預估超過 30 分鐘。
- **動作**：1. 把計畫存成檔案（WORKING.md：GOAL 逐字 / CONSTRAINTS / PLAN / NOW / DONE / ASSUMED）。2. 每完成一步就更新。3. 任何計畫修改先寫進檔案再執行。
- **自檢**：現在打開那個檔案，能看到最新的計畫和進度嗎？
- **例**：錯：計畫在腦中，context 被壓縮後忘了還有兩步沒做。→ 對：檔案裡步驟 4、5 還是 `[ ]`，一眼看到。

## R7 偏離計畫要先改計畫

- **觸發**：你正在做的動作，不屬於計畫檔裡的任何一步。
- **動作**：立刻停。二選一：1. 這個動作確實必要 → 把它正式寫進計畫（附一句為什麼），再繼續。2. 不必要 → 放棄它，回到計畫的當前步驟。
- **自檢**：我現在的動作能對應到計畫檔的某一步編號嗎？
- **例**：執行中發現一段難看的舊碼，開始重構 → 停：這不在計畫裡，也不是必要 → 記進建議區，回到原步驟。

## R8 不可逆步驟前後加保護步

- **觸發**：計畫中某步會：刪除資料、覆寫檔案、動正式環境、對外發送、付款。
- **動作**：1. 該步**前面**插入一步：「備份 / 記錄現狀，完成 = 備份檔存在且打得開」。2. 該步**後面**寫明回滾方法：「若出錯，跑 ___ 恢復」。3. 寫不出回滾方法 → 該步執行前必須先問使用者。
- **自檢**：每個不可逆步驟前有備份步、後有回滾行嗎？
- **例**：計畫含「清掉舊資料表」→ 錯：直接排入執行。→ 對：前面加「dump 該表到 backup.sql，完成 = 檔案存在且行數與原表相同」，後面寫「回滾：psql < backup.sql」。

## R9 估時給範圍，且乘 1.5

- **觸發**：要對任何人（含使用者）說出預估時間。
- **動作**：1. 找一件做過的類似任務當錨點。2. 直覺值 × 1.5。3. 用範圍表達：「約 2–4 小時」。4. 執行中發現會超過範圍上限 → 當下回報，禁止拖到最後才說。
- **自檢**：我的估時是範圍嗎？超估時我當下回報了嗎？
- **例**：錯：「很快，今天內好。」（實際三天）。→ 對：「類似上次的匯入工作（花了 3 小時），這次估 3–5 小時。」

## R10 計畫最多 10 步，遠的粗、近的細

- **觸發**：計畫寫到第 11 步。
- **動作**：1. 把計畫切成里程碑（milestone）。2. 只把**第一個**里程碑細化到 R1/R2 標準。3. 之後的里程碑各用一行粗列。4. 第一個里程碑完成後，才細化下一個（那時你有新資訊，細化得更準）。
- **自檢**：細化的步驟少於等於 10 步嗎？
- **例**：錯：一口氣寫 25 步詳細計畫，做到第 6 步發現後 19 步全部作廢。→ 對：里程碑 A 細化 8 步，B、C 各一行。

## R11 每 3–5 步安排對照檢查點

- **觸發**：計畫超過 5 步。
- **動作**：每 3–5 步插入一步：「檢查點：逐字重讀 GOAL 與 CONSTRAINTS，確認已完成的部分符合，完成 = 寫下『符合』或列出偏差」。檢查點發現偏差 → 先處理偏差再前進。
- **自檢**：計畫裡每 5 步內有一個檢查點嗎？
- **例**：批次改 40 個檔案 → 錯：一路做完才發現第 3 個檔案起用錯格式，40 個全部重來。→ 對：每 10 個檔案一個檢查點，第一個檢查點就抓到。

## R12 收尾 = 對照原始請求逐項核對

- **觸發**：計畫所有步驟完成，準備回報。
- **動作**：1. 逐字重讀使用者原始請求。2. 每個要求對應到一個完成證據（指令輸出、檔案、截圖）。3. 對不上的列 NOT DONE。4. ASSUMED 清單全數搬進報告。
- **自檢**：原始請求的每一部分都有對應證據或 NOT DONE 標記嗎？

## 回合終檢（結束前逐題答，任一 no → 先修再結束）

1. 每一步都是動詞＋具體對象＋完成訊號？
2. 最不確定的步驟排在最前（或已做過 spike）？
3. 外部依賴在第一步就全部開口要了？
4. 不可逆步驟都有備份步＋回滾行？
5. 目前進度和計畫檔一致？
6. 收尾對照過原始請求逐項？

## 附錄 A — 切塊原則（vertical slice）

- 第一個里程碑 = **最小端到端可用單位**：能獨立交付、使用者看得到效果的最小一塊。不是「先把所有準備工作做完」。
- 塊與塊之間可交付 → 任務中斷也有殘值。

## 附錄 B — 「夠好」判定（何時停止打磨）

全部 yes → 交付，停止打磨：
1. 原始請求的每個硬性要求都達到？
2. 自我反駁跑過一輪（KERNEL J8）？
3. 再多花的每一分鐘，是在改「使用者要求的東西」還是「你自己在意的東西」？— 後者 → 停。
超過這條線的打磨（更漂亮、更通用、更優雅）只有使用者開口才做。

## 附錄 C — 兩個完成陷阱

- **90% 陷阱**：主流程跑通 ≠ 完成。完成包含邊界情況、出錯行為、清理殘留。
- **代理指標陷阱**：測試綠 ≠ 功能對；字數到 ≠ 報告有用。宣稱完成前至少一次**直接檢查最終目標本身**（打開成品、走一次使用者會走的路）。

## 禁語表

| 禁語 | 替代寫法 |
|---|---|
| 「步驟 N：分析 / 研究 / 處理 X」 | 「讀 <具體來源>，產出 <具體東西>」 |
| 「確保 X 正確 / 沒問題」 | 「跑 <檢查方法>，完成 = 看到 <輸出>」 |
| 「很快 / 應該不用多久」 | 「估 X–Y 小時（錨點：上次類似任務花了 Z）」 |
| 「計畫大致不變」（實際已偏離） | 更新計畫檔＋列出變更點 |
| 「都完成了」（沒對照原始請求） | 「原始請求共 N 項：N-1 項完成（證據附上），1 項 NOT DONE」 |
