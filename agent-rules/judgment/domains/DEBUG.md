# DEBUG — 除錯規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：任何「東西壞了、修好它」的任務（不限程式：設定、pipeline、自動化皆同）。
> 每條規則三段：觸發（可觀察）→ 動作（可執行）→ 自檢（yes/no）。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。本檔沒寫到的情況，用 KERNEL.md。
> 規則依防災重要性排序，衝突時取編號小的。

## R1 修之前先重現

- **觸發**：收到「壞了 / 修 bug / 不能動」類任務。
- **動作**：1. 取得或建構一條能展示失敗的指令。2. 親自跑它——**在回報者的環境條件下**：locale、shell 與版本、cwd、PATH、使用者身分；拿不到就把「我的環境 vs 他的環境」差異列成假設清單逐項試。3. 把失敗輸出逐字存進工作筆記。4. 試 3 次仍無法重現 → 停止修理，回報「CANNOT REPRODUCE」＋三次嘗試的指令與**環境條件**，問使用者要重現步驟。
- **自檢**：我有一條本回合親自跑過、確實失敗的指令嗎？跑它的環境跟回報者的一致嗎？
- **例**：「登入壞了」→ 錯：直接去改 auth 程式碼。→ 對：先 `curl -X POST /api/login -d '...'` 看到 `401 Unauthorized`，存下來，才開始查。
- **例（環境變因）**：`--self-test` 在你的 shell 全綠、回報者說會炸 → 錯：判定對方誤報。→ 對：比對環境，`LC_ALL=zh_TW.UTF-8` 重跑 → 炸了——bug 是真的，只在中文 locale 觸發（bash 3.2 把全形字元吸進變數名）。「重現了指令」不等於「重現了環境」。

## R2 錯誤訊息逐字複製

- **觸發**：畫面上出現任何錯誤輸出。
- **動作**：複製貼上全文（含 stack trace 每一行）到工作筆記。禁止憑記憶重打或改寫。
- **自檢**：筆記裡的錯誤和終端機輸出逐字相同嗎？
- **例**：看到 `KeyError: 'user_id'` → 錯：筆記寫成「找不到 user 欄位的錯」。→ 對：原樣貼上。改寫後 `user_id` 變「user 欄位」，搜尋程式碼時就找不到了。

## R3 錯誤要讀到最後一行

- **觸發**：錯誤輸出超過 1 行。
- **動作**：1. 從最後一行往回讀到第一行。2. 找出第一個「屬於本專案檔案」的位置（不是 library 內部）。3. 打開那個檔案的那一行。
- **自檢**：我打開過 stack trace 指到的專案檔案了嗎？
- **例**：第一行寫 `ConnectionError`，最後三行寫 `config.yaml not found` → 錯：去查網路。→ 對：讀到最後發現是設定檔路徑錯。

## R4 先懷疑最近的改動

- **觸發**：之前能動的東西現在壞了。
- **動作**：1. 跑 `git log --oneline -10` 和 `git diff HEAD~3`（或翻出最近改過什麼）。2. 對每個最近改動問：「它碰到的檔案和錯誤位置有交集嗎？」3. 有交集的最先查。
- **自檢**：動手修之前，我看過最近的改動清單了嗎？
- **例**：昨天還能跑，今天報 import 錯誤 → 錯：開始查 Python 版本。→ 對：`git diff` 發現昨晚 rename 了一個模組。

## R5 一次只改一處

- **觸發**：要動手改任何東西來驗證想法。
- **動作**：1. 改一處。2. 跑 R1 的重現指令。3. 記錄結果。4. 沒修好 → **先把這個改動還原**，再試下一個想法。
- **自檢**：這次跑重現指令之前，距上次跑只改了一個地方嗎？
- **例**：懷疑 timeout 也懷疑編碼 → 錯：兩個一起改，過了，但不知道哪個是原因、另一個改動留成垃圾。→ 對：先只改 timeout，測；無效，還原；再改編碼。

## R6 禁止掩蓋式修法

- **觸發**：你的修法是以下任一：加 try/except（吞掉錯誤）、加 `if x is not None` 跳過、加 sleep / retry、放寬驗證條件、或任何「錯誤不再出現，但你說不出為什麼會發生」的改法。
- **動作**：停。先寫出一句話：「根本原因是 ___，證據在 <檔案:行號>」。寫不出來 → 回去查，禁止提交這個修法。寫得出來且掩蓋式修法確實是正解（例如外部服務真的會偶發斷線）→ 可以用，但把那句話寫進回報。
- **自檢**：我能一句話說出根本原因＋指出證據位置嗎？
- **例**：測試偶爾失敗 → 錯：加 retry 3 次，綠了。→ 對：查出兩個測試共用一個暫存檔沒清，修共享狀態。

## R7 修完跑原指令＋周邊測試

- **觸發**：你認為修好了。
- **動作**：1. 重跑 R1 那條指令，親眼看到成功。2. 跑該模組的測試（或手動走一次相鄰功能）。3. 把兩者的輸出逐字貼進回報。缺任何一項 → 回報寫「NOT VERIFIED: <缺什麼>」。4. 根因確認後，清點它的污染範圍（生效多久、弄壞了哪些資料/下游）→ 照 domains/CASCADE.md R1。
- **自檢**：回報裡有本回合跑出來的成功輸出嗎？
- **例**：錯：「改好了，應該可以了。」→ 對：「VERIFIED: `pytest tests/auth/ -x` 全綠（輸出如下），原重現指令回 200。」

## R8 三振停手

- **觸發**：對同一個錯誤，第 3 種不同修法失敗（改同一個參數的不同值算同一種）。
- **動作**：1. 禁止第 4 次嘗試。2. 還原所有實驗改動。3. 照抄模板回報：`STUCK:` 一句話 ／ `TRIED:` 三次的做法＋假設＋結果 ／ `ERROR(逐字):` ／ `HYPOTHESIS:` 目前最佳猜測 ／ `NEED:` 一個具體問題。
- **自檢**：我數了目前是第幾次失敗嗎？
- **例**：錯：第 5 次嘗試，越改越大範圍。→ 對：第 3 次失敗即產出 STUCK 報告 — 報告本身就是合格交付物。

## R9 API 行為用讀的，不用記的

- **觸發**：你的修法依賴某個函式、參數、設定的行為，而你本回合沒讀過它的定義或文件。
- **動作**：1. 打開定義（原始碼或官方文件），讀 signature 和說明。2. 找不到 → 在回報中寫「找不到 X 的定義」，禁止假設它存在或猜測其行為。
- **自檢**：這個修法用到的每個 API，我本回合都看過定義嗎？
- **例**：錯：記得 `requests` 預設 timeout 30 秒，據此推理。→ 對：查文件，發現預設是 `None`（無限等待），推理完全不同。

## R10 提交前重讀完整 diff

- **觸發**：準備回報修好了。
- **動作**：1. 產出完整 diff 並從頭讀一遍（是讀實際 diff，不是回想你改了什麼）。2. 刪掉：debug print、註解掉的舊碼、失敗實驗的殘留、與修復無關的改動。3. 再跑一次 R7。
- **自檢**：最終 diff 的每一行都是這次修復必需的嗎？
- **例**：diff 裡留著三行 `print("HERE")` 和一段註解掉的舊邏輯 → 全刪，重測，才交。

## R11 指控框架前先做最小重現

- **觸發**：你想說「這是 library / framework / 環境的 bug」。
- **動作**：寫一個 20 行以內、不含任何專案程式碼的腳本，重現出同樣錯誤。1. 重現成功 → 可以指控，附上腳本。2. 重現失敗 → 問題在你的程式碼，回到 R4。
- **自檢**：我有一個不含專案碼的最小重現腳本嗎？
- **例**：錯：「pandas 的 merge 有 bug。」→ 對：寫 10 行純 pandas 腳本測 — merge 正常，原來是自己的資料有重複 key。

## 回合終檢（結束前逐題答，任一 no → 先修再結束）

1. 有一條本回合跑過、先紅後綠的重現指令？
2. 回報裡的錯誤訊息全部是逐字複製？
3. 每輪測試之間只有一個變因？
4. 說得出根本原因＋證據位置？
5. 最終 diff 讀過一遍且無實驗殘留？
6. 每個「修好了」旁邊都貼了本回合的成功輸出？

## 禁語表

| 禁語 | 替代寫法 |
|---|---|
| 「應該修好了」「理論上可以了」 | 「VERIFIED: <指令> 轉綠（輸出如下）」或「NOT VERIFIED: 因為 ___」 |
| 「可能是環境問題」 | 「我檢查了 <具體設定>，值是 <值>（證據）」 |
| 「這是框架的 bug」（無最小重現） | 附 20 行最小重現腳本，或不說 |
| 「順便也修了 / 優化了一下」 | 不准順便。寫「另外注意到 X，需要的話我可以處理」 |
| 「大概是 race condition / 快取問題」 | 「我在 <檔案:行號> 看到 <具體證據>」，或標 GUESS |
