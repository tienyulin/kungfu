# ARCHITECT — 系統架構設計規則系統

> 讀者：任何模型（規則以 7B 等級為下限設計——弱模型照字面執行，強模型當檢查清單）。
> 適用：系統設計、架構評估、技術選型、重構規劃、容量規劃。
> 每條規則三段：觸發（可觀察）→ 動作（可執行）→ 自檢（yes/no）。
> 優先序：使用者最新明確訊息 > 專案 CLAUDE.md > 本檔 > 你的直覺。本檔沒寫到的情況，用 KERNEL.md。
> 規則依防災重要性排序，衝突時取編號小的。
> 本領域特殊性：**架構錯誤要一年後才發作，且發作時最貴**。所以規則全部作用在「現在就能檢查」的代理物上：數字、場景、記錄。

## R1 — 先要數字，沒有數字不設計

- **觸發**：收到「設計 / 架構 / 選型」類任務。
- **動作**：動筆前填這張表，缺的欄位先問使用者；問不到就寫 `ASSUMED: <值>（依據 ___）`：

| 項目 | 值 |
|---|---|
| 使用者數 / 併發數（現在、一年後） | |
| 資料量與成長率 | |
| 尖峰 QPS / 吞吐 | |
| 延遲要求（p95） | |
| 可接受停機時間 | |
| 團隊人數與技能 | |
| 預算（開發＋每月維運） | |

- **自檢**：設計文件第一節就是這張表嗎？每個空格都有值或 ASSUMED？
- **例**：「幫我設計一個電商後端」→ 錯：直接畫微服務全家桶。→ 對：先問「多少使用者？」答案是 500/日 → 一台 VM＋monolith＋RDBMS 就是正確答案。

## R2 — 先盤點現狀，不在真空裡設計

- **觸發**：目標系統已存在（重構、擴充、遷移），或團隊已有既定技術棧。
- **動作**：1. 列出現有元件、資料存放處、對外介面（讀 code / 設定 / 文件，不是聽轉述）。2. 列出團隊已經會運維的技術。3. 新設計裡每引入一個團隊沒運維過的技術，都要在文件中標「新技術：學習與維運成本 ___」。
- **自檢**：我的設計裡每個元件，都標了「既有」或「新引入」嗎？
- **例**：錯：團隊全是 PostgreSQL 經驗，設計丟了一個 Cassandra 進去因為「寫入量大」（實際寫入 200/s，PostgreSQL 單機就能吃）。→ 對：既有技術先算能不能撐，數字撐不住才引入新的。

## R3 — 從最無聊的方案開始，每個零件都要有數字背書

- **觸發**：開始畫方案。
- **動作**：1. 第一版一律是最無聊的：單體 ＋ 關聯式資料庫 ＋ 一層快取都沒有。2. 每加一個零件（queue、cache、微服務拆分、search engine、K8s），必須指出 R1 表裡**哪個數字**證明無聊版撐不住。指不出 → 不加。3. 「為了未來擴展性」不是數字。未來的量寫進 R1 表才算數字。
- **自檢**：每個零件都能一句話說出「它解掉哪個數字」嗎？
- **例**：錯：「加 Kafka 解耦比較好」（QPS 30）。→ 對：QPS 30 → 資料庫表當 queue 或直接同步呼叫；Kafka 等 QPS 表明需要時再進場。

## R4 — 資料先於服務

- **觸發**：方案裡出現超過一個存放資料的地方，或要拆分服務。
- **動作**：畫任何服務框之前先回答：1. 核心實體是哪些、誰**擁有**哪份資料（一份資料一個擁有者）。2. 哪些操作必須在同一個交易裡？——它們不能被拆到兩個服務。3. 資料被複製到第二個地方（cache、search index、另一服務）→ 寫下同步方式和不一致窗口多長、誰受影響。4. 既有資料要怎麼遷到新結構（沒有遷移計畫的新 schema 是紙上談兵）。
- **自檢**：每份資料都有唯一擁有者？每條交易邊界都畫在單一服務內？
- **例**：錯：訂單服務和庫存服務拆開，下單扣庫存跨服務又想要強一致 → 上線後靠分散式交易補救，複雜度爆炸。→ 對：交易邊界分析先做，要強一致就不拆，或改設計成可接受最終一致（超賣補償）。

## R5 — 用具體場景走查設計，一步一步走

- **觸發**：方案初稿完成。
- **動作**：挑 2 個最高頻場景＋1 個最極端場景（尖峰、大檔、批次），在設計圖上**逐步**走一遍：這個請求進來 → 打到哪個元件 → 讀哪份資料 → 回應路徑。每一步寫出協定與同步/非同步。走不下去或說不清的那一步，就是設計的洞。
- **自檢**：三個場景都留下逐步走查紀錄了嗎？
- **例**：走查「使用者上傳 500MB 檔案」→ 發現設計裡 API gateway timeout 30s → 洞找到了：需要分塊上傳或 presigned URL，在畫圖階段修掉，不是上線後。

## R6 — 每個零件問三次：死了、慢了、資料錯了會怎樣

- **觸發**：方案裡的每一個元件（含第三方服務）。
- **動作**：填故障表：每個元件 × 三種故障（完全掛掉／變慢 10 倍／回傳錯資料）→ 影響範圍、怎麼發現（監控什麼指標）、怎麼恢復。填完圈出單點故障清單；每個單點二選一：加冗餘，或寫一行「接受此風險，因為 ___」讓使用者簽收。
- **自檢**：故障表覆蓋了每個元件？單點清單使用者看過了？
- **例**：錯：Redis 當快取兼 session store，掛了全站登出＋資料庫被打穿——圖上沒人問過「Redis 死了會怎樣」。→ 對：故障表逼出這格，決定 session 進 DB、快取失效走限流。

## R7 — 信任邊界要畫出來

- **觸發**：設計中有任何外部輸入（使用者、第三方 API、上傳檔案）或多租戶。
- **動作**：1. 在圖上標出不可信輸入進入系統的每個點。2. 每個進入點寫：驗證什麼、認證在哪、授權在哪。3. 密鑰放哪裡（不是「環境變數」三個字帶過，是哪個 secret 管理機制）。4. 內部服務之間信不信任彼此？寫明。
- **自檢**：從任何外部輸入點出發，到達資料庫之前至少經過一次驗證與授權檢查嗎？
- **例**：錯：「內網服務不用驗證」→ 一台被打穿全部淪陷。→ 對：圖上畫出信任邊界，邊界內互信要寫成一條被簽收的風險。

## R8 — 演進路徑比終態圖重要

- **觸發**：設計的是既有系統的改造，或方案要分階段落地。
- **動作**：1. 交付物必須含「從現狀到終態」的分步計畫，每步獨立可上線、可回退（strangler 式），第一步兩週內能出貨。2. 「砍掉重練」是單向門 → 必須呈給使用者選，附兩案成本比較。3. 一次性檢查單向門：資料格式、對外 API 契約、ID 規則——這幾樣改起來最貴，設計時多想一步；其他的不為想像中的未來加料（R3）。
- **自檢**：計畫的第一步兩週內可上線？每步可回退？
- **例**：錯：「新系統寫好後一次切換」（寫了八個月，需求早變了）。→ 對：先把一個讀路徑導到新服務，灰度驗證，逐步搬。

## R9 — 每個重大決定留一筆 ADR

- **觸發**：做出技術選型或結構性決定（框架、資料庫、拆分方式、通訊協定）。
- **動作**：照 signals/CHOICES.md 模板寫 4 行：`DECIDED / BECAUSE（引用 R1 的數字）/ REJECTED（另一案＋一句為什麼不）/ REVISIT-IF（什麼數字出現時重看）`。
- **自檢**：BECAUSE 裡有數字嗎？REJECTED 有至少一個真實選項嗎？
- **例**：`DECIDED: PostgreSQL。BECAUSE: 資料 50GB、QPS<500、團隊熟。REJECTED: DynamoDB——查詢模式未定，鎖死 access pattern 太早。REVISIT-IF: 單表破 5 億列或跨區寫入需求出現。`

## R10 — 每個框一句話職責，每條線寫明是什麼

- **觸發**：畫方案圖、或 review 別人的圖。
- **動作**：1. 每個框寫一句話職責（「負責 ___」）；一句話裡出現「和」超過一次 → 這個框職責不清，拆或重想。2. 每條線標注：協定（HTTP/gRPC/queue）、同步或非同步、傳什麼、失敗語義（重試？丟棄？死信？）。3. 說不出來的框和線 = 還沒想清楚的部分，標成虛線並列入 open questions。
- **自檢**：圖上有沒有任何一條沒有標注的線？
- **例**：錯：兩個框之間一條裸箭頭（上線後才發現一邊以為同步一邊做成 fire-and-forget，掉單）。→ 對：線上標「HTTP 同步，timeout 3s，失敗回 503 由呼叫方重試」。

## R11 — 算維運帳：誰半夜被叫醒、每月多少錢

- **觸發**：方案定稿前。
- **動作**：1. 列出方案要人維運的東西（升級、備份、監控、on-call），對照 R1 的團隊人數——維運面積超過團隊胃納 → 換 managed service 或砍零件。2. 估每月帳單（雲資源、授權、SaaS），數量級就好，寫進文件。3. 每個元件寫「掛了誰會知道」——沒有監控訊號的元件等於沒設計完。
- **自檢**：維運清單、月費估算、監控訊號三樣都在文件裡嗎？
- **例**：錯：2 人團隊上自管 K8s ＋ 自架 Kafka ＋ 自架 ES（全職都在餵基礎設施）。→ 對：managed 等價物或砍掉，人力留給業務。

## R12 — 交付前紅隊自己：說出最先壞的三個地方

- **觸發**：設計文件要交付。
- **動作**：1. 寫下「這個設計最可能先壞的三個地方」＋各自的訊號和備案。寫不出三個 = 你還沒認真找（每個設計都有至少三個）。2. open questions 單獨成節，不准藏進內文。3. 交付語氣照證據分級：走查過的部分用肯定句，沒走查的標「未驗證」。
- **自檢**：文件裡有「最先壞的三個地方」這一節嗎？

## 回合終檢（結束前逐題答，任一 no → 先修再結束）

1. 文件開頭有數字表，每格有值或 ASSUMED？
2. 每個零件都指得出它解哪個數字？
3. 三個場景走查＋故障表都做了？
4. 資料擁有者與交易邊界畫清了？
5. 有演進路徑，第一步兩週內可上線？
6. ADR、單點風險簽收、最先壞的三處，都在文件裡？

## 禁語表

| 禁語 | 替代寫法 |
|---|---|
| 「為了未來的擴展性」 | 「R1 表的 ___ 數字顯示 12 個月內會到 ___，所以 ___」 |
| 「微服務比較好維護」 | 指出哪個數字（團隊數、部署頻率、模組衝突）證明單體撐不住 |
| 「這是業界標準做法」 | 「它解我們的 ___ 問題，證據是 ___」（別人的標準解別人的量級） |
| 「安全之後再加」 | 信任邊界現在畫，R7 |
| 「資料庫細節之後再說」 | 資料先於服務，R4 |
| 「理論上可以撐到百萬用戶」 | 「走查過 ___ 場景到 ___ 量級；再往上未驗證」 |
