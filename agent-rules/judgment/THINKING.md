# THINKING — 思考框架（產出答案的迴圈）

> 一套通用的思考程序。讀者：任何模型。
> 與 KERNEL.md 的分工：KERNEL 是**約束**（防止做錯），本檔是**生成**（怎麼想出答案）。
> 用法：預估 >30 分鐘或高風險的問題跑完整迴圈；小問題只跑 FRAME ＋ ATTACK。
> 每一步都有可判定的出口條件——不確定自己在哪一步，就從 FRAME 重來。

---

## 思考迴圈（六步，依序）

### 1. FRAME — 定框

1. 用自己的話重述問題一句（不是複述原文，是翻譯成你要交付什麼）。
2. 把手上資訊分三堆：**已知**（有證據）／**未知**（可以去查）／**假設**（暫時當真，要標記）。假設堆裡的每一項都寫下來——思考出錯多半是假設堆裡混進了以為是已知的東西。
3. 判斷問題類型，決定主武器：

| 問題類型 | 主武器 |
|---|---|
| 事實問題（是什麼、多少） | 查證，證據分級（KERNEL J2） |
| 選擇問題（選哪個） | signals/CHOICES.md → 深水區 signals/TRADEOFFS.md |
| 故障問題（為什麼壞） | 回溯＋重現（domains/DEBUG.md） |
| 設計問題（怎麼做） | 約束先行：先列硬限制再想方案 |
| 創作問題（寫/畫/做東西） | 先定成功標準：好的長什麼樣、給誰 |

4. 最重要的一問：「**真正要解的是這個嗎？**」——被叫來修 X 時，X 常常只是 Y 的症狀（KERNEL J1）。
- **出口條件**：能寫出「我要交付 ___，已知 ___，關鍵未知是 ___」三格。寫不出 → 資訊不夠，先問或先查。

### 2. SPREAD — 展開

1. 承諾任何方向之前，先生成 **2–3 條不同的路**（廣度優先一層，不用深）。
2. 機械規則：**寫下第 2 個選項之前，不准開始執行第 1 個。** 第一個想法的品質是隨機的——它只是最先到，不是最好。
3. 生成第 2、3 條路的技巧：換一個立場想（使用者怎麼解？維運怎麼解？）、反轉問題（見橫向動作）、把第 1 條路的核心假設否定掉再想。
- **出口條件**：紙上有 ≥2 條真正不同的路（攻擊不同假設，不是同路的參數變體）。

### 3. PROBE — 試探

1. 對每條路，找出「它成立所依賴的最可疑假設」。
2. 排序下一步檢查：**便宜 × 排除力**——先做「花費最小、但做完能砍掉最多選項」的那個檢查。
3. 一次一個變因（KERNEL J5），每次試探後更新三堆（已知/未知/假設）。
- **出口條件**：剩下一條明顯領先的路；或所有路都死了 → 回 SPREAD 用新資訊重新展開。

### 4. COMMIT — 收斂

1. 收斂判準：**剩餘選項之間的預期差異 < 再查一輪的成本** → 停止研究，選了。
2. 選定後寫一行為什麼（signals/CHOICES.md 的 4 行模板），然後全力執行，不再回頭懷疑——懷疑留給第 5 步集中處理。
- **出口條件**：有一個被寫下來的決定＋理由。

### 5. ATTACK — 自攻

1. 對自己的答案問三個問題：
   - 「什麼情況下這是錯的？」寫出至少一條具體破綻，**實際去檢查它**。
   - 「立場相反的人會攻擊哪裡？」（最強反方論點，不是稻草人）。
   - 「如果這是錯的，最可能錯在哪一步？」回去看那一步的證據等級。
2. 破綻檢查不了 → 交付時標明「未檢查：___」。
- **出口條件**：至少一條破綻被實際檢查過（KERNEL J8）。

### 6. DELIVER — 交付

結論先行、證據分級決定語氣、假設堆全數列出（KERNEL J9/J2/J7、domains/WRITING.md）。

## 橫向動作（任何一步卡住時，選一個換檔）

| 動作 | 做法 | 什麼時候用 |
|---|---|---|
| **換高度** | 上一層問「為什麼要做這個」；下一層找「最小具體案例」 | 原地打轉、細節淹沒 |
| **換表述** | 同一問題換一種說法重寫一遍（換領域語言、換視角） | 讀三遍還是沒感覺 |
| **極端測試** | 把關鍵變數推到 0 和無限大，看結論還成不成立 | 檢驗一個聽起來合理的論證 |
| **反轉** | 想不出「怎麼成功」→ 改列「怎麼必然失敗」，然後逐條避開 | 設計、規劃、風險分析 |
| **具體化** | 抽象爭論不下去 → 找一個具體實例實際跑跑看 | 兩個抽象方案吵不出高下 |
| **類比檢查** | 類比可以用來產生假設，但用之前必須寫出「本案與類比案的一個關鍵差異」 | 每次想說「這就像 ___」的時候 |

## 心智常數（校準值，直接背）

1. **第一個解釋通常不對，但方向常常有用**——留下它的線索，丟掉它的結論。
2. **兩個異常同時出現 → 先假設同一個根因**，兩個獨立巧合的機率遠低於你的直覺。
3. **答案越流暢，越要防幻覺**——流暢感來自語言，不來自正確性。
4. **意外的觀察比預期的觀察值錢**——跟預測不符的那個數據點是最大的資訊來源，別把它當雜訊排掉。
5. **別人都走過的路走不通，先問「為什麼他們沒成功」再走**——你大概率不是第一個想到的人。
6. **簡單解釋先驗證**——複雜解釋常常只是給不了解的東西編故事。
7. **想說「應該只是巧合」的時候，再查一次**——這句話是大腦想下班的訊號。

## 微型示範（完整迴圈 60 秒版）

問題：「資料庫每天凌晨 3 點變慢。」

- **FRAME**：交付 = 根因＋修法。已知：時間固定。未知：3 點在跑什麼。假設：跟排程有關（標記，未證實）。類型 = 故障問題。
- **SPREAD**：路 1 = 我們自己的排程（備份/批次）；路 2 = 資料庫內建維護（stats 收集）；路 3 = 外部流量（別的時區尖峰、爬蟲）。
- **PROBE**：最便宜高排除力 = 看 3 點的活動紀錄（一次查詢排除兩條路）→ 發現備份 job 與報表批次重疊。
- **COMMIT**：根因 = IO 競爭。修法 = 錯開排程（可逆、零成本，先做）。
- **ATTACK**：破綻——「重疊」是相關不是因果。檢查：錯開一天，觀察是否消失 → 消失，因果成立。
- **DELIVER**：「根因：備份與報表批次重疊搶 IO（證據：錯開後恢復正常）。已錯開排程。假設過『與排程有關』已證實。」
