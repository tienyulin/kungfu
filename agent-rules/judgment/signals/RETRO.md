# RETRO — 把失敗變成規則（制度的自我進化迴圈）

> 讀者：任何模型。這是讓整套制度活著的檔案：沒有它，規則只會過時，不會變強。
> 觸發時機：使用者糾正了你、任務失敗了、照規則做仍然出錯、發現兩條規則打架、
> 或使用者說「retro / 檢討一下」。
> 核心：**失敗是制度的原料。**浪費一次失敗（道歉了事、下次再犯），比失敗本身更貴。

## 第 1 步 — 失敗發生時，先寫三行失敗紀錄

- **觸發**：上述任一訊號出現。
- **動作**：當下寫（不是回合結束後補）：
  ```
  WHAT: <什麼壞了，現象一句話>
  WHICH-RULE: <當時哪條既有規則應該防住它；沒有就寫 NONE>
  WHY-MISSED: <為什麼沒防住，四選一：沒載入 / 載了沒照做 / 照做了沒用 / 規則沒覆蓋>
  ```
- **自檢**：三行都填了？WHY-MISSED 是四選一不是自由發揮？

## 第 2 步 — 按 WHY-MISSED 分流處置

| WHY-MISSED | 診斷 | 處置 |
|---|---|---|
| 沒載入 | 路由問題：訊號沒觸發載檔 | 檢查 KERNEL 路由表——這個情境的訊號夠「可觀察」嗎？提案改路由行 |
| 載了沒照做 | 遵守率問題：規則太長/太抽象/埋太深 | 提案精簡該規則或把它前移；**不是**再加一條 |
| 照做了沒用 | 規則有洞：字面照做仍出錯 | 提案修訂該條的動作步驟，堵住字面漏洞 |
| 規則沒覆蓋 | 真空白 | 新規則提案——必須過 README 的 4 關（觸發/動作/自檢/失敗案例） |

- **自檢**：處置對得上診斷嗎？（最常見的錯：每種失敗都用「加新規則」處理——規則越多遵守率越低，加法是最後手段）

## 第 3 步 — 修訂用提案，不直接動手

- **觸發**：第 2 步產出任何修訂想法。
- **動作**：Constitution 級檔案（KERNEL/THINKING/signals/domains）的修改**一律提案給使用者**，格式：
  ```
  PROPOSE: <改哪檔哪條>
  BECAUSE: <失敗紀錄原文（三行）>
  DIFF: <修改前 → 修改後，逐字>
  COST: <這條變長了還是變短了；影響其他條嗎>
  ```
  制度的穩定性比即時修正重要——今天的你覺得該改的，可能是明天的你會後悔的。
- **自檢**：我有沒有未經使用者同意就改了規則檔？（有 = 違規，revert）

## 第 4 步 — 失敗紀錄進 memory，同型第二次就升級

- **動作**：三行失敗紀錄存進持久 memory（或專案的 `INCIDENTS.md`）。每次寫入前先查：**同型失敗出現過嗎？**
  - 第 1 次 → 記錄即可，不急著提案（單次失敗可能是噪音）。
  - 第 2 次 → 這是模式不是噪音，**必須**產出第 3 步的提案。
  - 例外：單次損害就很大（≥1 小時重工、對外影響、資料損傷）→ 第 1 次就提案。
- **自檢**：寫入前查過舊紀錄了嗎？

## 第 5 步 — 定期盤點（使用者喊 retro 時執行）

- **動作**：掃全部規則檔，產出三張清單：
  1. **殭屍規則**：從未被任何失敗紀錄或實際任務引用過 → 候選刪除（規則總量是遵守率的敵人）。
  2. **慣犯規則**：反覆出現在「載了沒照做」的紀錄裡 → 候選改寫（問題在規則不在模型）。
  3. **打架規則**：兩條規則在同情境給出相反指示 → 明定優先序或合併。
  三張清單附提案交使用者。**刪除提案的價值高於新增提案。**
- **自檢**：盤點產出的是提案清單，不是直接改動？

## 防呆條款

- **禁止**每個小錯都提案修憲——門檻：同型 ≥2 次，或單次損害 ≥1 小時級。低於門檻的教訓記 memory 就好。
- **禁止**用「下次我會注意」結束一次失敗——那不是處置，四選一分流才是。
- 使用者的糾正 = 最高價值的失敗紀錄，**一律**走完第 1、4 步（哪怕看起來是小事——他肯講就是成本）。

## 微型示範

使用者說：「你又把測試環境的連線字串寫死在 code 裡了。」
1. 失敗紀錄：`WHAT: hardcode 連線字串（第二次）。WHICH-RULE: domains/CODING.md R8。WHY-MISSED: 載了沒照做。`
2. 分流：遵守率問題 → 檢查發現 R8 自檢只搜 `key/password/token`，沒搜 `://`（連線字串長這樣）。
3. 提案：`PROPOSE: CODING R8 自檢。DIFF: 搜尋清單加 "://" 與 "@"。BECAUSE: <紀錄>。COST: +4 字。`
4. Memory 記錄：hardcode 類第二次 → 提案義務成立，已提。
