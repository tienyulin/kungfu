# LONGRUN — 長任務的失憶保護協議

> 觸發時機：任務預估超過 30 分鐘、或超過 5 個步驟、或跨 session。
> 核心事實：**你的 context 會被壓縮或丟失，磁碟不會。**
> 你對「我剛剛在做什麼」的記憶不可信；寫在檔案裡的才可信。

## 開工規則

符合觸發條件 → 動工前先在工作目錄建 `WORKING.md`：

```markdown
# WORKING — <任務名>（<絕對日期>）

## GOAL（使用者原話，逐字貼上，禁止改寫）
> <原始請求全文>

## CONSTRAINTS（使用者提過的所有限制，隨對話持續補充）
- <限制 1>

## PLAN
1. [ ] <步驟：具體到「跑 X / 改 Y / 查 Z」>
2. [ ] ...

## NOW
<正在做 PLAN 的第幾步、做到哪>

## DONE
- <完成的步驟 + 一句話結果>

## ASSUMED
- <所有假設，格式同 J7>

## TOUCHED
- <動過的檔案 / 系統清單>
```

## 運行規則

1. **每完成一個 PLAN 步驟**就更新 WORKING.md（不是每個動作 — 那太吵）。
2. **每 5 步做一次漂移檢查**：逐字重讀 GOAL 和 CONSTRAINTS，回答「我現在的動作是 PLAN 的第幾步？」答不出來 = 已漂移 → 停，要嘛正式改 PLAN（寫下為什麼改），要嘛回到 PLAN 上。
3. **計畫改變必須寫進 PLAN**，不准只在腦中改。腦中的改動活不過一次 context 壓縮。
4. **CONSTRAINTS 隨時補**：使用者在對話中間丟出的任何限制（「不要動 X」「用中文」「預算 Y」），當下就寫進去。對話早期的限制是最容易被你遺忘的東西。

## 恢復規則（session 重啟 / context 被壓縮後）

1. 先讀 WORKING.md，再讀使用者原始請求，**然後才准動手**。
2. 對照 TOUCHED 清單實際檢查現場狀態（檔案真的改了嗎？），不要相信 DONE 清單的記憶 — 用現場證據校正它。
3. 發現現場狀態和 WORKING.md 對不上 → 以現場為準，更新 WORKING.md，回報差異。

## 交付前規則

- 逐字重讀 GOAL + CONSTRAINTS 最後一次，對照交付物（J10）。
- ASSUMED 清單全數搬進最終報告。

## 收尾規則

- 任務結束 → WORKING.md 刪除，或改名 `DONE-<日期>-<任務名>.md` 歸檔（有長期參考價值才留）。
- **禁止**把密鑰、token、個資寫進 WORKING.md — 它會被 commit、被同步、被讀走。

## 為什麼這些規則存在（讓你願意遵守）

- 弱模型長對話的兩大死法：**忘記早期限制**、**不知不覺偏離目標**。兩者當事模型都毫無自覺，所以解法只能是機械性的外部檢查點，不能靠「我會注意」。
- WORKING.md 同時是你的救命繩和使用者的進度窗：他隨時打開就知道你在哪、卡在哪。
