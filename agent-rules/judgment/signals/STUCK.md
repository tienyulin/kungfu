# STUCK — 卡關偵測與脫困程序

> 觸發時機：同一問題失敗 2 次的當下（第 3 次嘗試之前）就要讀本檔。
> 核心：卡關不可恥；在迴圈裡燒掉一小時而不自知才可恥。好的卡關報告是成果。

## 迴圈偵測訊號（任一出現 = 你已經在迴圈裡）

1. 同一個錯誤訊息出現第 2 次。
2. 你正在復原自己 20 分鐘前做的改動。
3. 你對現象的解釋越來越長、越來越曲折（真相通常越查越簡單）。
4. 你開始怪罪工具 / 框架 / 環境「有 bug」（可能，但機率排最後，見下方基率表）。
5. 你想再試「跟剛剛差不多」的第 3 種變體。

## 三振規則（硬性）

同一障礙 3 次**真正不同**的修法失敗 → 停，禁止第 4 次。

「真正不同」= 攻擊不同的假設。改參數 1→2→3 是**一次**嘗試的三個變體，不是三次嘗試。

停下後照抄模板：

```
STUCK: <一句話說卡在哪>
TRIED:
  1) <做法> — 因為 <假設> — 結果 <逐字現象>
  2) ...
  3) ...
EVIDENCE (逐字): <錯誤訊息 / 關鍵輸出，複製貼上>
BEST HYPOTHESIS: <目前最佳猜測 + 為什麼還沒驗證它>
NEED: <一個具體問題或一個需要的東西>
```

## 換高度清單（第 2 次失敗後，換一招而不是再撞一次）

依序考慮，選一個執行：

1. **縮小重現**：把問題縮到最小案例。一半以上的問題在縮小過程中自己現形。
2. **重讀原始錯誤**：逐字重讀第一個錯誤訊息的**每一行**，包括你當時跳過的部分。答案常在你略過的那行。
3. **假設盤點**：寫下「我目前相信為真但沒驗證過的事」清單，逐一驗證。卡關的原因幾乎總在這張清單裡。特別檢查**表面相似陷阱**：這個問題「看起來像」你認得的問題，於是套了認得的解法——寫出本案例和記憶模式的一個差異點，寫不出來 = 你沒真的看這個案例。
4. **二分法**：能用排除法把問題空間砍半就砍半（停用一半、回到已知好的狀態再逐步加回）。
5. **質疑題目**：我在修的是正確的問題嗎？會不會現象是 A、我卻一直在修 B？回到最初的現象重新描述一次。
6. **從乾淨狀態重來**：復原所有實驗性改動（J5），從已知好的狀態重新出發。實驗殘留常常就是新問題的來源。

## 嫌疑人基率表（先查機率高的）

出問題時，按這個順序懷疑，不准跳級：

1. **我剛剛改的東西**（機率最高，永遠第一個查）
2. 我的假設錯了（環境、版本、設定跟我以為的不同）
3. 輸入資料跟預期不同
4. 設定檔 / 環境差異
5. 相依套件的已知問題（查它的 issue tracker）
6. 工具 / 框架本身的 bug（機率極低，需要最小重現才准指控）

「一定是框架的 bug」出現在第 1 步之前 = 你在逃避查自己的改動。

## 復原紀律

- 每個失敗的假設，其改動**立刻復原**，再試下一個。
- 失敗實驗堆疊 3 層之後，你已經不知道現在的現象是原始問題還是實驗殘留造成的。
