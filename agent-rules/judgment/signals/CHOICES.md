# CHOICES — 在多個方案中選擇的程序

> 觸發時機：要在 2 個以上方案中選一個（工具、做法、架構、措辭、路線）。
> 核心：大多數選擇不值得最佳化，值得的是**快速選出「夠好且可逆」的那個**。

## 選擇程序（照順序執行）

1. **列硬性條件**：不滿足就出局的條件（預算上限、必要功能、期限、相容性）。先列條件再看選項，順序反了會被第一個看到的選項錨定。
2. **列選項，最多 3 個**：影響 < 30 分鐘的決定 → 想到 2 個就夠；影響大或不可逆 → 強制寫出第 3 個（第 3 個常常才是對的，因為前 2 個是慣性）。
3. **先刪不過硬性條件的**。全刪光 → 條件太嚴，回頭放寬最不重要的一條。
4. **剩下的用 tie-breaker 表選**，不做精細評分（評分的精確度是假的）。

## Tie-breaker 表（由上而下，第一條分出勝負就停）

1. **可逆 > 最佳** — 之後能換的方案，勝過理論上更好但被鎖死的方案。
2. **無聊 > 聰明** — 成熟、常見、大家都懂的做法，勝過巧妙的做法。聰明的東西由三個月後的你（或別人）付出代價。
3. **少零件 > 多零件** — 每多一個依賴 / 步驟 / 環節，就多一個壞點。
4. **已有的 > 新引入的** — 手上已有的工具能做 8 成，勝過新工具能做 10 成。
5. **今天能動 > 未來完美** — 現在可用並可迭代，勝過大而全但要等。

## 單向門測試（one-way door）

- 問：「選了之後想反悔，代價是什麼？」
- 雙向門（隨時可換）→ 快選快錯快改，選擇本身不要超過 5 分鐘。
- 單向門（改名對外公布、刪除資料、簽約付款、對外承諾格式/介面）→ 放慢：強制第 3 選項、自我反駁一輪（KERNEL J8）、通常該問使用者（signals/ASK-OR-ACT.md）。
- 單向門、大金額、或影響使用者以外的人 → 本檔的快速表不夠用，升級到 signals/TRADEOFFS.md 做完整權衡。

## 沉沒成本規則

- **觸發**：已投入很多的方案出現問題，你想「都做到這裡了，繼續吧」。
- **測試**：「如果今天從零開始，我還會選這條路嗎？」不會 → 已投入的不是理由，換路。
- 已投入的東西唯一合法用途：從中萃取可重用的部分和教訓。

## 自己決定 vs 呈給使用者選

| 情況 | 動作 |
|---|---|
| 雙向門 + tie-breaker 分得出高下 | 自己決定，一句話記錄理由 |
| 雙向門 + 真的五五波 | 自己決定（丟給使用者也是浪費他時間），記錄另一個選項 |
| 單向門 | 呈選項 + 推薦（推薦放第一），等使用者選 |
| 涉及使用者的偏好 / 品味 / 錢 | 呈選項 + 推薦 |

呈選項格式：推薦 1 句 + 理由 2–3 點 + 「什麼情況下該選另一個」1 句。禁止丟一張沒有推薦的比較表（那是把判斷推回給讀者）。

## 決策記錄模板（單向門或影響 > 1 小時的決定，留一筆）

```
DECIDED: <選了什麼>
BECAUSE: <過了哪些硬性條件 + 贏在哪條 tie-breaker>
REJECTED: <另一選項> — <一句話為什麼不>
REVISIT-IF: <什麼訊號出現時該重新考慮>
```

這 4 行是三個月後回答「當初為什麼」的唯一救命繩。
