# VERIFY — 驗證配方（按成品類型查表）

> 觸發時機：任何東西要宣稱「完成 / 可用 / 正確」之前。
> 兩條鐵律：
> 1. **驗證必須有可能失敗。** 不可能失敗的檢查不是檢查（跑一個永遠會過的指令 = 沒驗）。
> 2. 驗不了就寫 `NOT VERIFIED - <原因>` + `TO VERIFY: <誰、用什麼方法可以驗>`。
> 3. **代理指標不算數**：測試綠 ≠ 功能對。清單跑完後，仍要直接看一眼最終目標本身。

## 程式碼改動

- [ ] 實際執行（指令 + exit code + 關鍵輸出，逐字留存）
- [ ] 相關測試跑過；沒有測試 → 手動走一次使用者會走的路徑
- [ ] 至少一個邊界輸入（空、超大、重複執行、非法值）
- [ ] 把**最終 diff 完整重讀一遍**（不是回憶你改了什麼，是讀實際的 diff）— 找實驗殘留、debug 輸出、多餘改動

## 設定 / 環境改動

- [ ] 改完後服務真的重新載入了（很多設定改了但沒生效，這是最常見的假完成）
- [ ] 一個行為探針：做一件**只有新設定生效才會成功**的事
- [ ] 記下舊值，寫在報告裡（回滾用）

## 文件 / 報告

- [ ] 以讀者身分從頭到尾讀一遍（不是掃過，是讀）
- [ ] 每個數字能指出來源；每個連結實際打開過
- [ ] 日期全部絕對值；「目前 / 最新」類字眼旁有日期
- [ ] 只保留第一句給讀者看，他知道結論（domains/WRITING.md R3）

## 研究結論

- [ ] 關鍵主張有第一手來源（原始碼、官方文件、原始數據），不是二手轉述
- [ ] 重要主張有 2 個獨立來源；矛盾照實呈現
- [ ] 信心用語與證據等級一致（全文一種語氣 = 沒分級；見 domains/RESEARCH.md R12）

## 對外訊息（email、留言、公開發文）

- [ ] 收件人 / 對象正確（寄錯人不可逆）
- [ ] 提到的附件真的附了；提到的連結是對的連結
- [ ] 人名、公司名、金額、日期逐字核對
- [ ] 沒有內部資訊、密鑰、不該外流的脈絡
- [ ] 全文唸一遍（抓漏字和語氣事故）→ 然後仍然先給使用者看（J3：對外 = 等同意）

## 資料處理（轉換、清洗、匯入匯出）

- [ ] 前後筆數對帳：輸入 N 筆 → 輸出應該幾筆？差額能逐筆解釋嗎？
- [ ] 隨機抽 3 筆，人工對照原始資料逐欄核對
- [ ] 一個聚合值 sanity check（總和、平均在合理數量級？）

## 刪除 / 清理

- [ ] 先列清單再動手：完整列出將刪除的項目 + 數量
- [ ] 清單和意圖對照：有沒有混進不該刪的？（一個都不確定 → 移出清單）
- [ ] 能進垃圾桶 / 備份就不要直接銷毀
- [ ] 刪完核對：刪掉的 = 清單上的，不多不少

## 自動化 / 排程（cron、hook、CI）

- [ ] 手動觸發一次完整流程，看到成功輸出
- [ ] 故意弄壞一次輸入，確認失敗時的行為可接受（會報錯？還是靜默吞掉？）
- [ ] 確認它下次真的會被觸發（排程有註冊、hook 有掛上）

## 完成宣告模板（照抄填空）

```
VERIFIED: <做了哪個驗證>
EVIDENCE: <逐字輸出 / 看到什麼，1–5 行>
NOT VERIFIED: <沒驗到的部分 + 原因>（沒有就寫 none）
```
